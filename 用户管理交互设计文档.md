# 用户管理交互设计文档

更新时间：2026-04-05

本文档定义「用户管理」模块的前后端交互规范，供开发实施参考。

## 一、模块定位

用户管理是 **SYSTEM_ADMIN 专属**功能，用于管理系统中所有用户账号。普通用户（USER 角色）无法看到或进入此模块。

用户管理是后续「项目组管理」和「成员管理」的前置依赖——只有系统里先有用户，才能把用户添加到项目组。

## 二、现有数据模型

当前 `wb_user` 表结构（V2 迁移）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT, 自增主键 | 用户 ID |
| `username` | VARCHAR(64), 唯一 | 登录名 |
| `password_hash` | VARCHAR(255) | 密码哈希（BCrypt） |
| `display_name` | VARCHAR(64) | 展示名 |
| `system_role` | VARCHAR(32) | `SYSTEM_ADMIN` 或 `USER` |
| `status` | VARCHAR(32) | `ACTIVE` 或 `DISABLED` |
| `last_login_at` | DATETIME | 最近登录时间 |
| `created_by` | BIGINT | 创建人 ID |
| `updated_by` | BIGINT | 更新人 ID |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

## 三、导航入口

### 菜单位置

在顶部导航栏追加「用户管理」入口，仅 `systemAdmin` 可见：

```
首页 | 数据源管理 | 自助查询 | 用户管理(仅SYSTEM_ADMIN)
```

### 路由

- 路径：`/users`
- 守卫：仅 `systemAdmin === true` 可进入；非系统管理员访问时显示 Unauthorized 页面
- 注意：用户管理不依赖项目组上下文（不需要 `RequireGroup`），它是系统级功能

## 四、用户列表页

### 页面结构

沿用数据源列表页的布局模式：

```
┌─────────────────────────────────────────────┐
│ [搜索输入框]                    [新建用户]     │
├─────────────────────────────────────────────┤
│ 用户名 | 展示名 | 系统角色 | 状态 | 最近登录 | 创建时间 | 操作 │
│ ─────────────────────────────────────────── │
│ admin  | admin | 系统管理员 | 已启用 | 2026-... | 2026-... | 编辑 禁用 重置密码 │
│ ...    |       |           |       |         |         |                    │
├─────────────────────────────────────────────┤
│ 本页 N 条，共 M 条          每页 [10条]  页码  │
└─────────────────────────────────────────────┘
```

### 表格列定义

| 列 | 字段 | 说明 |
|---|---|---|
| 用户名 | `username` | 登录名 |
| 展示名 | `displayName` | 显示名称 |
| 系统角色 | `systemRole` | 显示为标签：`系统管理员` / `普通用户` |
| 状态 | `status` | 状态药丸：绿色 `已启用` / 灰色 `已禁用` |
| 最近登录 | `lastLoginAt` | 格式 `YYYY-MM-DD HH:mm:ss`，无记录时显示 `从未登录` |
| 创建时间 | `createdAt` | 格式 `YYYY-MM-DD HH:mm:ss` |
| 操作 | — | 操作按钮组 |

### 搜索

- 单一搜索框，placeholder：`搜索用户名或展示名`
- 输入时防抖 300ms 后触发请求
- 搜索参数同步到 URL query（`?keyword=xxx`），刷新页面保留搜索状态
- 后端对 `username` 和 `display_name` 做模糊匹配

### 分页

- 默认每页 10 条
- 可选：10 / 20 / 50
- 分页参数同步到 URL（`?page=1&size=10`）
- 复用现有 `DataSourcePagination` 组件模式

### 操作按钮

每行根据条件显示：

| 操作 | 图标 | 条件 | 说明 |
|---|---|---|---|
| 编辑 | `Pencil` | 始终显示 | 打开编辑对话框 |
| 禁用 | `Ban` | 当 `status=ACTIVE` 且不是当前登录用户 | 禁用该用户 |
| 启用 | `CheckCircle` | 当 `status=DISABLED` | 启用该用户 |
| 重置密码 | `KeyRound` | 始终显示 | 重置该用户密码 |

**不提供「删除用户」操作**。原因：
- 用户可能已关联项目组成员、数据源负责人、审计日志等
- 删除会导致外键级联问题和数据追溯困难
- 「禁用」已经足够满足停用需求

## 五、新建用户

### 入口

列表页右上角「新建用户」按钮，点击打开对话框。

### 对话框结构

```
┌──── 新建用户 ──────────────── [×] ─┐
│                                    │
│  用户名 *                           │
│  [________________]                │
│                                    │
│  展示名 *                           │
│  [________________]                │
│                                    │
│  初始密码 *                          │
│  [________________] [👁]           │
│                                    │
│  系统角色                            │
│  [普通用户 ▾]                       │
│                                    │
│  ── 加入项目组（可选）──              │
│                                    │
│  项目组              项目组角色        │
│  [Alpha 项目组 ▾]   [开发者 ▾]  [×] │
│  [Beta 项目组 ▾]    [项目组管理员 ▾] [×] │
│                                    │
│  [+ 添加项目组]                      │
│                                    │
│           [取消]  [确认创建]          │
└────────────────────────────────────┘
```

### 字段规则

| 字段 | 必填 | 校验规则 | 说明 |
|---|---|---|---|
| 用户名 | ✅ | 2-64 字符，仅允许字母、数字、下划线、短横线；不可与已有用户名重复 | 创建后不可修改 |
| 展示名 | ✅ | 1-64 字符 | 用于页面显示 |
| 初始密码 | ✅ | 8-64 字符，至少包含字母和数字 | 带密码显示/隐藏切换 |
| 系统角色 | 否 | 下拉选择：`普通用户`（默认）/ `系统管理员` | — |
| 项目组分配 | 否 | 动态多行列表，每行包含项目组下拉 + 角色下拉 + 删除按钮 | 不添加则创建后用户不属于任何组 |

### 「加入项目组」区段说明

- 此区段用分隔线和标题 `加入项目组（可选）` 与上方基本信息区分
- 默认无任何行，显示「+ 添加项目组」按钮
- 点击「+ 添加项目组」新增一行：项目组下拉（placeholder `请选择项目组`）+ 角色下拉（默认 `开发者`）+ 删除按钮（×）
- 支持添加多行，每行独立选择项目组和角色
- **已选的项目组从其他行的下拉列表中排除**，防止重复选择同一项目组
- 点击行尾的 × 按钮删除该行
- 当所有项目组都已被选择时，「+ 添加项目组」按钮置灰不可点击
- 每行的项目组和角色都必须选择，未选完整的行在提交时标红提示
- 项目组列表数据来源：`GET /api/v1/groups`（返回所有项目组，仅 SYSTEM_ADMIN 可调用）

### 交互规则

1. 点击「确认创建」后按钮进入 loading 态，显示 `创建中...`
2. 创建成功：
   - 关闭对话框
   - 列表自动刷新
   - 如果指定了项目组：提示 `用户 xxx 创建成功，已加入 N 个项目组`
   - 如果未指定项目组：提示 `用户 xxx 创建成功`
3. 创建失败：
   - 用户名已存在 → 用户名输入框下方显示红色提示：`该用户名已被使用`
   - 其他错误 → 对话框底部显示红色错误提示
   - 不关闭对话框，不清空已填内容

### 前端校验（提交前）

- 所有必填字段非空，为空时对应输入框红色边框 + placeholder 提示（复用登录页模式）
- 用户名格式校验
- 密码强度校验
- 项目组行如果添加了但未选择项目组 → 该行项目组下拉标红
- 项目组行如果选了项目组但未选角色 → 该行角色下拉标红

## 六、编辑用户

### 入口

列表操作列「编辑」按钮。

### 对话框结构

```
┌──── 编辑用户 ──────────────── [×] ─┐
│                                    │
│  用户名                              │
│  admin （只读，不可编辑）              │
│                                    │
│  展示名 *                           │
│  [admin___________]                │
│                                    │
│  系统角色                            │
│  [系统管理员 ▾]                     │
│                                    │
│  ── 项目组管理 ──                    │
│                                    │
│  项目组              项目组角色        │
│  [Alpha 项目组 ▾]   [开发者 ▾]  [×] │
│  [Beta 项目组 ▾]    [项目组管理员 ▾] [×] │
│                                    │
│  [+ 添加项目组]                      │
│                                    │
│           [取消]  [保存修改]          │
└────────────────────────────────────┘
```

### 字段规则

| 字段 | 可编辑 | 说明 |
|---|---|---|
| 用户名 | ❌ | 只读展示，灰色文本 |
| 展示名 | ✅ | 同新建规则 |
| 系统角色 | ✅ | 下拉选择；但**不可修改自己的角色**（防止唯一管理员自降权） |
| 项目组分配 | ✅ | 动态多行列表，回填当前已加入的项目组及角色，支持增删改 |

### 「项目组管理」区段说明

- 此区段用分隔线和标题 `项目组管理` 与上方基本信息区分
- 打开编辑对话框时，调用 `GET /api/v1/users/{id}/groups` 获取该用户当前的项目组分配，回填为多行列表
- 如果用户当前不属于任何项目组，则显示空列表和「+ 添加项目组」按钮
- 交互规则与新建用户的「加入项目组」一致：
  - 点击「+ 添加项目组」新增一行
  - 已选的项目组从其他行的下拉列表中排除
  - 点击行尾 × 按钮删除该行（即从项目组中移除）
  - 可以修改已有行的角色（即变更项目组内角色）
- 保存时后端做 diff：对比提交的 `groupAssignments` 和数据库现有记录，执行新增、删除、角色变更

### 交互规则

1. 打开时自动填充当前值（基本信息 + 项目组列表）
2. 如果编辑的是当前登录用户，「系统角色」下拉框禁用，旁边显示灰色提示：`不可修改自己的角色`
3. 保存成功 → 关闭对话框 + 刷新列表 + 成功提示
4. 保存失败 → 对话框内显示错误，不关闭

### 关于密码

编辑对话框**不包含密码字段**。修改密码通过独立的「重置密码」操作完成。原因：
- 分离关注点，避免误操作
- 重置密码是高危操作，需要单独确认

## 七、禁用 / 启用用户

### 禁用

1. 点击「禁用」按钮
2. 弹出确认对话框：
   ```
   确定要禁用用户 admin 吗？
   禁用后该用户将无法登录系统，但其已有数据不会被删除。
   [取消]  [确认禁用]
   ```
3. 「确认禁用」按钮为红色/警告色
4. 确认后：
   - 调用后端接口修改状态
   - 成功 → 列表刷新 + 提示 `用户 xxx 已禁用`
   - 失败 → 提示错误信息

### 启用

1. 点击「启用」按钮
2. 无需确认对话框，直接调用接口
3. 成功 → 列表刷新 + 提示 `用户 xxx 已启用`

### 保护规则

- **不可禁用自己** — 当前登录用户的禁用按钮不显示
- 后端同样需要校验此规则

## 八、重置密码

1. 点击「重置密码」按钮
2. 弹出对话框：
   ```
   ┌──── 重置密码 ──────────────── [×] ─┐
   │                                    │
   │  为用户 admin 设置新密码             │
   │                                    │
   │  新密码 *                           │
   │  [________________] [👁]           │
   │                                    │
   │  确认密码 *                         │
   │  [________________] [👁]           │
   │                                    │
   │           [取消]  [确认重置]          │
   └────────────────────────────────────┘
   ```
3. 校验规则：

4. 确认后：
   - 成功 → 关闭对话框 + 提示 `密码已重置`
   - 失败 → 对话框内显示错误

## 九、后端 API 设计

所有接口均需 `Authorization: Bearer {token}`，且仅 `SYSTEM_ADMIN` 可调用。非 SYSTEM_ADMIN 调用返回 `403 Forbidden`。

### 9.1 用户列表（分页 + 搜索）

```
GET /api/v1/users?page=1&size=10&keyword=admin
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [
      {
        "id": 1,
        "username": "admin",
        "displayName": "管理员",
        "systemRole": "SYSTEM_ADMIN",
        "status": "ACTIVE",
        "lastLoginAt": "2026-04-04T17:00:00",
        "createdAt": "2026-04-01T10:00:00"
      }
    ],
    "total": 1,
    "size": 10,
    "current": 1,
    "pages": 1
  }
}
```

注意：响应**不包含** `passwordHash`（实体已有 `@JsonIgnore`）。

### 9.2 项目组列表（供下拉选择）

```
GET /api/v1/groups
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": [
    { "id": 1, "name": "alpha", "description": "Alpha 项目组" },
    { "id": 2, "name": "beta", "description": "Beta 项目组" }
  ]
}
```

仅 SYSTEM_ADMIN 可调用。返回所有项目组（不分页，项目组数量通常有限）。

### 9.3 创建用户

```
POST /api/v1/users
Content-Type: application/json

{
  "username": "new_user",
  "displayName": "新用户",
  "password": "Abc12345",
  "systemRole": "USER",
  "groupAssignments": [
    { "groupId": 1, "groupRole": "DEVELOPER" },
    { "groupId": 2, "groupRole": "GROUP_ADMIN" }
  ]
}
```

其中 `groupAssignments` 为可选数组。若提供，则每个元素的 `groupId` 和 `groupRole` 均为必填（`groupRole` 可选值：`GROUP_ADMIN` 或 `DEVELOPER`）。数组可以为空或不传，表示不加入任何项目组。

成功响应：`201 Created`，返回创建后的用户对象（不含密码）。

错误响应：
- 用户名已存在 → `409 Conflict`，`message: "用户名已存在"`
- 项目组不存在 → `400 Bad Request`，`message: "指定的项目组不存在"`
- 同一项目组出现多次 → `400 Bad Request`，`message: "不可重复指定同一项目组"`
- 参数校验失败 → `400 Bad Request`

后端逻辑：
- 校验用户名唯一
- 密码用 BCrypt 哈希后存储
- `createdBy` 设为当前登录用户 ID
- `status` 默认 `ACTIVE`
- 若提供了 `groupAssignments`：在同一事务中往 `wb_project_group_member` 批量插入成员记录

### 9.4 编辑用户

```
PUT /api/v1/users/{id}
Content-Type: application/json

{
  "displayName": "新展示名",
  "systemRole": "USER",
  "groupAssignments": [
    { "groupId": 1, "groupRole": "DEVELOPER" },
    { "groupId": 3, "groupRole": "GROUP_ADMIN" }
  ]
}
```

其中 `groupAssignments` 为可选数组，表示该用户编辑后应属于的全部项目组及角色。传空数组 `[]` 表示从所有项目组中移除。不传此字段则不修改项目组关系。

成功响应：`200 OK`，返回更新后的用户对象。

错误响应：
- 用户不存在 → `404 Not Found`
- 尝试修改自己的 systemRole → `400 Bad Request`，`message: "不可修改自己的系统角色"`
- 项目组不存在 → `400 Bad Request`，`message: "指定的项目组不存在"`
- 同一项目组出现多次 → `400 Bad Request`，`message: "不可重复指定同一项目组"`

后端逻辑：
- **不可修改** `username`（即使传了也忽略）
- **不可修改** `password`（密码通过独立接口重置）
- 校验不可自改角色
- `updatedBy` 设为当前登录用户 ID
- 若提供了 `groupAssignments`：在同一事务中做 diff 更新
  - 对比提交列表与数据库中该用户现有的 `wb_project_group_member` 记录
  - 新增的组 → 插入新记录
  - 删除的组 → 删除对应记录
  - 角色变更的组 → 更新 `role` 字段

### 9.5 修改用户状态（禁用 / 启用）

```
PATCH /api/v1/users/{id}/status
Content-Type: application/json

{
  "status": "DISABLED"
}
```

成功响应：`200 OK`。

错误响应：
- 禁用自己 → `400 Bad Request`，`message: "不可禁用自己"`
- 用户不存在 → `404 Not Found`

### 9.6 重置密码

```
POST /api/v1/users/{id}/reset-password
Content-Type: application/json

{
  "newPassword": "NewPass123"
}
```

成功响应：`200 OK`。

错误响应：
- 密码不符合规则 → `400 Bad Request`
- 用户不存在 → `404 Not Found`

### 9.7 获取用户项目组列表

```
GET /api/v1/users/{id}/groups
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": [
    { "groupId": 1, "groupName": "Alpha 项目组", "groupRole": "DEVELOPER" },
    { "groupId": 2, "groupName": "Beta 项目组", "groupRole": "GROUP_ADMIN" }
  ]
}
```

返回该用户当前所属的所有项目组及其在组内的角色。用于编辑用户时回填项目组列表。仅 SYSTEM_ADMIN 可调用。

## 十一、实施建议

### 后端

1. 新建 `UserController`（`/api/v1/users`）
2. 新建 `UserService`（业务逻辑：校验、密码哈希、权限检查）
3. 新建请求/响应 DTO：`CreateUserRequest`、`UpdateUserRequest`、`ResetPasswordRequest`、`UserResponse`
4. 在 Controller 层做 SYSTEM_ADMIN 权限校验（可复用现有 token 解析逻辑）

### 前端

1. 新建 `src/api/user.ts` — API 层
2. 新建 `src/views/UserList.tsx` — 列表页（参照 DataSourceList）
3. 新建 `src/views/users/UserTable.tsx` — 表格组件
4. 新建 `src/views/users/UserForm.tsx` — 新建/编辑对话框
5. 新建 `src/views/users/ResetPasswordDialog.tsx` — 重置密码对话框
6. 修改 `Layout.tsx` — 添加导航入口（`systemAdmin` 条件）
7. 修改 `router/index.tsx` — 添加路由（仅 systemAdmin 守卫，不需要 RequireGroup）

### 建议开发顺序

1. 后端 API（全部 5 个接口）
2. 前端列表页 + 搜索 + 分页
3. 新建用户
4. 编辑用户
5. 禁用/启用
6. 重置密码
7. 联调 + 浏览器验证
