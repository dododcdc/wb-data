# 项目组角色业务权限边界 V1

更新时间：2026-04-01

本文档用于在"用户 / 项目组 / 角色 V1 方案"基础上，进一步明确项目组内两个角色在各业务模块中的真实能力边界。

关联总纲文档：

- [用户-项目组-角色-V1讨论纪要.md](/Users/wenbin/Projects/wb-data/用户-项目组-角色-V1讨论纪要.md)

## 一、本文档的目标

总纲文档已经确认了以下内容：

- 系统层角色：`SYSTEM_ADMIN` / `USER`
- 项目组层角色：`GROUP_ADMIN` / `DEVELOPER`
- 用户可以属于多个项目组
- 同一个用户在不同项目组下可以有不同角色
- 权限是统一来源，前后端共用一套 permission 定义

本文档不再讨论这些总原则，而是只回答一个问题：

**在具体业务模块里，这两个项目组角色分别能做什么，不能做什么？**

## 二、角色范围

本文档只讨论项目组内角色：

- `GROUP_ADMIN`
- `DEVELOPER`

不单独展开 `SYSTEM_ADMIN`，因为 `SYSTEM_ADMIN` 默认拥有所有项目组全部权限。后续如有特殊例外，再单独补充。

## 三、讨论方法

每个模块统一按以下几个动作维度讨论：

- 查看
- 创建
- 编辑
- 删除
- 执行
- 导出
- 管理他人相关内容

如果某个模块不涉及某类动作，可以明确写"无此动作"。

### 通用原则：无创建者特权

V1 不实现"创建者特权"机制。即：

- 共享资源（数据源、离线任务等）的操作权限完全由角色决定，不因"谁创建的"而有额外权限
- 例如：即使某人创建了一条记录，也不能因此获得超出其角色的编辑/删除权限
- 好处：权限模型简单一致，避免"创建者能改但同角色其他人不能改"的混乱局面
- 后续版本如有需要，可以在此基础上扩展细粒度的所有权控制

## 四、模块一：数据源管理

### 需要讨论的问题

1. 谁能看到数据源列表
2. 谁能看到数据源详情
3. 谁能创建数据源
4. 谁能编辑数据源
5. 谁能删除数据源
6. 谁能测试连接
7. 谁能看到敏感字段
   - 用户名
   - 密码是否可回显
   - 连接参数是否全部可见
8. 谁能修改"负责人"之类的元信息
9. 是否允许非管理员编辑自己创建的数据源
10. 数据源删除是否需要额外限制
    - 是否必须无依赖
    - 是否必须只有管理员可删

### 角色能力矩阵（已确认）

| 动作 | GROUP_ADMIN | DEVELOPER | 备注 |
|---|---|---|---|
| 查看列表 | ✅ | ✅ | 两类角色可见，由 `datasource.read` 控制 |
| 查看详情 | ✅ | ✅ | 两类角色可进入详情页 |
| 创建 | ✅ | ❌ | 仅 GROUP_ADMIN，由 `datasource.write` 控制 |
| 编辑 | ✅ | ❌ | 同上 |
| 删除 | ✅ | ❌ | 同上；V1 建议：有离线任务依赖时禁止删除 |
| 测试已有数据源连接 | ✅ | ✅ | 针对已保存的数据源做连通性验证 |
| 测试编辑中配置连接 | ✅ | ❌ | 属于配置编辑过程的一部分 |
| 查看敏感信息 | ✅ | ❌ | 密码永不回显；用户名/连接参数仅 GROUP_ADMIN 可见完整值，DEVELOPER 脱敏显示 |

### 边界说明

1. **密码永不回显** — 任何角色（包括 GROUP_ADMIN）在详情页都看不到明文密码。编辑时密码字段为空，提交空值表示不修改
2. **敏感信息分级** — 用户名和连接参数对 GROUP_ADMIN 完整展示，对 DEVELOPER 脱敏（如 `u***r`）或隐藏。具体脱敏策略可实施时再定
3. **详情页对两类角色开放，但写操作仅限 GROUP_ADMIN** — DEVELOPER 可以看详情和做已有连接测试，但不能修改配置
4. **测试连接分两类** — 对“已保存的数据源”做连通性验证，与“编辑中配置”的测试分开讨论。后者属于配置编辑过程的一部分，仅 GROUP_ADMIN 可用
5. **无创建者特权** — 不存在"谁创建谁才能编辑"的逻辑，参见§三通用原则
6. **删除前置检查** — V1 建议删除数据源时检查是否有离线任务依赖，如有则拒绝删除并提示
7. **元信息修改**（如负责人字段）— V1 暂不单独设计"负责人"字段；如有需要，编辑权限跟随 `datasource.write`

### 当前已知前提

- 数据源属于项目组共享资源
- 只有 `GROUP_ADMIN` 可以创建/编辑/删除本组数据源（`datasource.write` 仅 GROUP_ADMIN 拥有）
- `GROUP_ADMIN` 和 `DEVELOPER` 都可以进入数据源详情页（已确认）
- `DEVELOPER` 不能创建、编辑、删除数据源（已确认，与现有数据平台一致）
- `DEVELOPER` 可以在列表操作列测试“已有数据源连接”（已确认）

## 五、模块二：自助查询

### 需要讨论的问题

1. 谁能进入自助查询页面
2. 谁能执行 SQL
3. 谁能格式化 SQL
4. 谁能查看表结构/元数据
5. 谁能导出查询结果
6. 谁能使用图钉功能
7. 谁能查看自己的导出任务
8. 谁能查看别人的导出任务
9. 谁能查看未来可能存在的查询历史
10. 是否限制某些角色只能查询不能导出

### 角色能力矩阵（已确认）

| 动作 | GROUP_ADMIN | DEVELOPER | 备注 |
|---|---|---|---|
| 进入查询页 | ✅ | ✅ | 由 `query.use` 控制 |
| 执行 SQL | ✅ | ✅ | 同上 |
| 格式化 SQL | ✅ | ✅ | 纯前端功能，跟随页面可见性即可 |
| 查看元数据（表结构） | ✅ | ✅ | 跟随 `query.use`，能查询就能看表结构 |
| 导出结果 | ✅ | ✅ | 由 `query.export` 控制 |
| 使用图钉 | ✅ | ✅ | 跟随 `query.use`，图钉是个人记录 |
| 查看自己的导出任务 | ✅ | ✅ | 个人记录，只要有 `query.export` 就能看自己的 |
| 查看他人的导出任务 | ❌ | ❌ | 导出任务是个人记录，任何角色都不能看他人的 |

### 边界说明

1. **当前双角色方案下，两类项目组角色都可进入查询模块** — V1 不再区分“只读查询角色”
2. **查询与导出是两个独立权限** — 总纲中 `query.use` 和 `query.export` 分开定义。但 V1 中这两个权限的角色映射完全一致（GA/DEV 都有），所以 V1 实际上不存在"能查询但不能导出"的角色。如果未来需要限制某角色只能查不能导，调整权限矩阵即可
3. **格式化 SQL 不需要独立权限** — 这是纯前端文本处理，不涉及后端，跟随页面可见性
4. **图钉是个人记录** — 每个用户只能操作自己的图钉，不存在"管理他人图钉"的动作
5. **导出任务完全隔离** — 即使 GROUP_ADMIN 也不能查看其他成员的导出任务
6. **查询历史（V1 不做）** — 如果未来实现，归属个人记录 + 项目组上下文，规则同导出任务

### 当前已知前提

- `query.use` 与 `query.export` 是两类不同 permission
- 导出任务属于"个人记录 + 项目组上下文"
- 项目组管理员默认不能查看他人的个人记录
- `GROUP_ADMIN / DEVELOPER` 都可以执行 SQL（已确认）
- `GROUP_ADMIN / DEVELOPER` 都可以导出结果（已确认）
- 导出任务仅本人可见，任何角色都不能查看他人的导出任务（已确认）
- 图钉仅本人可见和可操作（已确认）

## 六、模块三：离线开发

### 需要讨论的问题

1. 谁能看到离线开发模块
2. 谁能查看离线任务列表
3. 谁能查看任务详情
4. 谁能创建离线任务
5. 谁能编辑离线任务
6. 谁能删除离线任务
7. 谁能运行/重跑任务
8. 谁能停止任务
9. 谁能查看执行日志
10. 谁能查看任务产物

### 角色能力矩阵（已确认）

| 动作 | GROUP_ADMIN | DEVELOPER | 备注 |
|---|---|---|---|
| 进入模块 | ✅ | ✅ | 由 `offline.read` 控制 |
| 查看任务列表 | ✅ | ✅ | 同上 |
| 查看任务详情 | ✅ | ✅ | 同上 |
| 创建任务 | ✅ | ✅ | 由 `offline.write` 控制 |
| 编辑任务 | ✅ | ✅ | 同上；可编辑任何人创建的任务（无创建者特权） |
| 删除任务 | ✅ | ✅ | 同上；V1 建议：运行中的任务不可删除 |
| 运行/重跑任务 | ✅ | ✅ | 由 `offline.write` 控制 |
| 停止任务 | ✅ | ✅ | 同上 |
| 查看日志/产物 | ✅ | ✅ | 由 `offline.read` 控制 |

### 边界说明

1. **当前双角色方案下，两类项目组角色都可进入离线开发模块** — V1 不再单独拆分“分析师”或“只读角色”
2. **离线任务是共享资源** — GROUP_ADMIN 和 DEVELOPER 都能看到并操作本组所有离线任务，不限于自己创建的
3. **无创建者特权** — DEVELOPER A 创建的任务，DEVELOPER B 也可以编辑/运行/停止/删除
4. **GROUP_ADMIN 与 DEVELOPER 权限一致** — 在离线开发模块中，两个角色的能力完全相同（都有 `offline.read` + `offline.write`）
5. **运行中的任务不可删除** — V1 建议：任务状态为"运行中"时，删除按钮禁用，需先停止再删除
6. **日志和产物跟随读权限** — 有 `offline.read` 即可查看任何任务的执行日志和产物

### 当前已知前提

- 离线任务/脚本/产物属于项目组共享资源
- `offline.read` 与 `offline.write` 是两类不同 permission
- 当前总纲文档里默认：
  - `GROUP_ADMIN` 可管理离线任务
  - `DEVELOPER` 可使用离线开发

## 七、模块四：成员管理

### 需要讨论的问题

1. 谁能查看成员列表
2. 谁能查看成员角色
3. 谁能添加成员（从已有用户中选择）
4. 谁能移除成员
5. 谁能修改成员角色
6. 是否可以修改自己的角色
7. 最后一个 GROUP_ADMIN 的保护规则

### 角色能力矩阵（已确认）

| 动作 | GROUP_ADMIN | DEVELOPER | 备注 |
|---|---|---|---|
| 查看成员列表 | ✅ | ✅ | 由 `member.read` 控制，两类角色可见 |
| 查看成员角色 | ✅ | ✅ | 同上，角色信息随列表展示 |
| 添加成员 | ✅ | ❌ | 由 `member.manage` 控制；只能从已有用户中选择，不能创建新用户 |
| 移除成员 | ✅ | ❌ | 同上；不能移除最后一个 GROUP_ADMIN |
| 修改成员角色 | ✅ | ❌ | 同上；可以将成员提升为 GROUP_ADMIN |
| 修改自己的角色 | ❌ | ❌ | 任何人不能修改自己的角色（防止自我提权或意外降级） |

### 边界说明

1. **成员列表对两类项目组角色可见** — 知道"谁在组里"是基本协作需求
2. **添加成员 ≠ 创建用户** — GROUP_ADMIN 只能从系统已有用户中搜索并添加到本组，创建新用户是 SYSTEM_ADMIN 的事
3. **最后一个 GROUP_ADMIN 保护** — 如果组内只剩一个 GROUP_ADMIN，该成员不可被移除、不可被降级。需由 SYSTEM_ADMIN 介入处理
4. **不可自我修改角色** — 防止 GROUP_ADMIN 误操作自降为 DEVELOPER，也防止 DEVELOPER 自我提权
5. **移除成员的数据处理** — 参见总纲§八：个人记录保留但不可见，共享资源不受影响

### 当前已知前提

- `member.read` 两类角色都有
- `member.manage` 仅 GROUP_ADMIN 拥有
- GROUP_ADMIN 可以把组内成员提升为 GROUP_ADMIN（总纲已确认）
- 最后一个 GROUP_ADMIN 不可自行降级或退出（总纲已确认）
- 两类角色都可以查看成员列表和角色信息（已确认）
- 只有 GROUP_ADMIN 可以添加成员、移除成员、修改成员角色（已确认）
- 任何人都不能修改自己的角色（已确认）

## 八、模块五：项目组设置

### 需要讨论的问题

1. 谁能查看项目组基本信息（名称、描述）
2. 谁能修改项目组名称
3. 谁能修改项目组描述
4. 谁能删除项目组

### 角色能力矩阵（已确认）

| 动作 | GROUP_ADMIN | DEVELOPER | 备注 |
|---|---|---|---|
| 查看项目组信息 | ✅ | ✅ | 基本信息所有成员可见（名称、描述在导航栏/选择器中已展示） |
| 修改名称 | ✅ | ❌ | 由 `group.settings` 控制 |
| 修改描述 | ✅ | ❌ | 同上 |
| 删除项目组 | ❌ | ❌ | 项目组删除是系统级操作，仅 SYSTEM_ADMIN 可执行 |

### 边界说明

1. **项目组基本信息人人可见** — 项目组名称和描述在导航栏项目组选择器中就会展示，无需额外权限
2. **设置修改仅限 GROUP_ADMIN** — 项目组名称和描述的修改由 `group.settings` 控制
3. **删除项目组超出项目组角色范围** — 总纲已确认：只有 SYSTEM_ADMIN 可以删除项目组，且只能删除空项目组（无成员、无数据源）。这不是项目组内角色的能力
4. **V1 设置项较少** — 目前项目组设置只有名称和描述。后续如增加更多设置（如默认数据源、通知配置等），统一跟随 `group.settings` 权限

### 当前已知前提

- `group.settings` 仅 GROUP_ADMIN 拥有
- 项目组删除是 SYSTEM_ADMIN 的系统级操作，不在本矩阵讨论范围内
- V1 只允许删除无成员、无数据源的空项目组（总纲已确认）
- 两类角色都可以查看项目组基本信息（已确认）
- 只有 GROUP_ADMIN 可以修改项目组名称和描述（已确认）
- V1 的项目组设置范围仅包含名称和描述（已确认）
