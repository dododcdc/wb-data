# 项目组管理交互设计文档

更新时间：2026-04-05

本文档定义系统级「项目组」页面的前后端交互规范，供开发实施参考。

## 一、模块定位

项目组管理是 **SYSTEM_ADMIN 专属**功能，位于顶部导航「系统管理」下拉菜单中。用于在系统层面管理所有项目组。

本页面只做三件事：
1. **查看**所有项目组列表
2. **创建**新项目组
3. **删除**空项目组

**不在此页面做的事**：编辑项目组名称/描述、管理成员——这些在组级「成员与设置」页面完成。

## 二、现有数据模型

当前 `wb_project_group` 表结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT, 自增主键 | 项目组 ID |
| `name` | VARCHAR(64), 唯一约束 | 项目组名称 |
| `description` | VARCHAR(255) | 项目组描述 |
| `created_by` | BIGINT | 创建人 ID |
| `updated_by` | BIGINT | 更新人 ID |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

关联表 `wb_project_group_member`：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT, 自增主键 | 记录 ID |
| `group_id` | BIGINT | 项目组 ID |
| `user_id` | BIGINT | 用户 ID |
| `role` | VARCHAR(32) | `GROUP_ADMIN` 或 `DEVELOPER` |
| `created_by` | BIGINT | 创建人 ID |
| `updated_by` | BIGINT | 更新人 ID |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

## 三、导航入口

### 菜单位置

在顶部导航栏新增「系统管理」下拉菜单（仅 `systemAdmin` 可见），项目组作为其二级菜单项：

```
普通用户：    首页 | 数据源管理 | 自助查询 | 成员与设置
SYSTEM_ADMIN：首页 | 数据源管理 | 自助查询 | 成员与设置 | 系统管理 ▾
                                                         ├ 用户管理
                                                         └ 项目组
```

### 路由

- 路径：`/groups`
- 守卫：仅 `systemAdmin === true` 可进入；非系统管理员访问时显示 Unauthorized 页面
- 注意：项目组管理不依赖项目组上下文（不需要 `RequireGroup`），它是系统级功能

## 四、项目组列表页

### 页面结构

沿用数据源列表页和用户列表页的布局模式：

```
┌─────────────────────────────────────────────┐
│ [搜索输入框]                     [新建项目组]  │
├─────────────────────────────────────────────┤
│ 项目组名称 | 描述 | 成员数 | 创建时间 | 操作    │
│ ─────────────────────────────────────────── │
│ alpha     | Alpha 项目组 | 5  | 2026-... | 删除 │
│ beta      | Beta 项目组  | 3  | 2026-... | 删除 │
│ ...       |             |    |          |      │
├─────────────────────────────────────────────┤
│ 本页 N 条，共 M 条          每页 [10条]  页码   │
└─────────────────────────────────────────────┘
```

### 表格列定义

| 列 | 字段 | 说明 |
|---|---|---|
| 项目组名称 | `name` | 项目组唯一标识名 |
| 描述 | `description` | 项目组描述，无内容时显示 `—` |
| 成员数 | `memberCount` | 该项目组的成员总数 |
| 创建时间 | `createdAt` | 格式 `YYYY-MM-DD HH:mm:ss` |
| 操作 | — | 操作按钮组 |

### 搜索

- 单一搜索框，placeholder：`搜索项目组名称`
- 输入时防抖 300ms 后触发请求
- 搜索参数同步到 URL query（`?keyword=xxx`），刷新页面保留搜索状态
- 后端对 `name` 做模糊匹配

### 分页

- 默认每页 10 条
- 可选：10 / 20 / 50
- 分页参数同步到 URL（`?page=1&size=10`）
- 复用现有分页组件模式

### 操作按钮

每行根据条件显示：

| 操作 | 图标 | 条件 | 说明 |
|---|---|---|---|
| 删除 | `Trash2` | 当 `memberCount === 0` | 仅空项目组（无成员）可删除 |

**当项目组有成员时**，删除按钮不显示。

**不提供「编辑」操作**。原因：
- 项目组名称和描述的修改在组级「成员与设置」页面完成，由 GROUP_ADMIN 负责
- 系统级页面职责单一：只管创建和删除

## 五、新建项目组

### 入口

列表页右上角「新建项目组」按钮，点击打开对话框。

### 对话框结构

```
┌──── 新建项目组 ─────────────── [×] ─┐
│                                     │
│  项目组名称 *                        │
│  [________________]                 │
│                                     │
│  描述                                │
│  [________________]                 │
│  [________________]                 │
│                                     │
│           [取消]  [确认创建]           │
└─────────────────────────────────────┘
```

### 字段规则

| 字段 | 必填 | 校验规则 | 说明 |
|---|---|---|---|
| 项目组名称 | ✅ | 2-64 字符，仅允许字母、数字、下划线、短横线；不可与已有项目组名重复 | 唯一标识，表有 `uk_wb_project_group_name` 约束 |
| 描述 | 否 | 最长 255 字符 | 多行文本输入（textarea），可为空 |

### 交互规则

1. 点击「确认创建」后按钮进入 loading 态，显示 `创建中...`
2. 创建成功：
   - 关闭对话框
   - 列表自动刷新
   - 提示 `项目组 xxx 创建成功`
3. 创建失败：
   - 项目组名称已存在 → 名称输入框下方显示红色提示：`该项目组名称已被使用`
   - 其他错误 → 对话框底部显示红色错误提示
   - 不关闭对话框，不清空已填内容

### 前端校验（提交前）

- 项目组名称非空，为空时输入框红色边框 + placeholder 提示
- 项目组名称格式校验（字母、数字、下划线、短横线，2-64字符）
- 描述长度校验（≤255字符）

### 创建后的状态

新建项目组后：
- 项目组没有任何成员（空项目组）
- SYSTEM_ADMIN 需要在「用户管理」编辑用户时将用户分配到该项目组，或在组级「成员与设置」页面添加成员
- 空项目组可以被删除

## 六、删除项目组

### 入口

列表操作列「删除」按钮（仅当 `memberCount === 0` 时显示）。

### 交互规则

1. 点击「删除」按钮
2. 弹出确认对话框：
   ```
   确定要删除项目组 alpha 吗？
   删除后不可恢复。
   [取消]  [确认删除]
   ```
3. 「确认删除」按钮为红色/警告色
4. 确认后：
   - 调用后端接口删除项目组
   - 成功 → 列表刷新 + 提示 `项目组 xxx 已删除`
   - 失败 → 提示错误信息

### 删除前置条件（后端校验）

- 项目组必须没有任何成员（`wb_project_group_member` 中无记录）
- V1 暂不检查数据源依赖（后续可扩展：有数据源时也不可删除）
- 后端需校验 memberCount === 0，不满足则返回 `400 Bad Request`

## 七、后端 API 设计

所有接口均需 `Authorization: Bearer {token}`，且仅 `SYSTEM_ADMIN` 可调用。非 SYSTEM_ADMIN 调用返回 `403 Forbidden`。

### 7.1 项目组分页列表（扩展现有接口）

现有 `GET /api/v1/groups` 返回全量列表。需要改造为支持分页、搜索和成员数统计：

```
GET /api/v1/groups?page=1&size=10&keyword=alpha
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [
      {
        "id": 1,
        "name": "alpha",
        "description": "Alpha 项目组",
        "memberCount": 5,
        "createdAt": "2026-04-01T10:00:00"
      }
    ],
    "total": 2,
    "size": 10,
    "current": 1,
    "pages": 1
  }
}
```

**兼容性考虑**：现有不分页的 `GET /api/v1/groups` 在「用户管理」的项目组下拉选择中使用。两种方案：
- **方案A**：通过请求参数区分 — 不传 `page`/`size` 时返回全量列表（保持向后兼容），传了则返回分页结果
- **方案B**：保留现有接口用于下拉选择，新增独立路径用于列表页

建议采用**方案A**，减少接口数量。后端检测到 `page` 参数时走分页逻辑，否则返回全量简单列表。

### 7.2 创建项目组

```
POST /api/v1/groups
Content-Type: application/json

{
  "name": "new_group",
  "description": "新项目组的描述"
}
```

其中 `description` 为可选字段。

成功响应：`201 Created`，返回创建后的项目组对象。

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 3,
    "name": "new_group",
    "description": "新项目组的描述",
    "memberCount": 0,
    "createdAt": "2026-04-05T15:30:00"
  }
}
```

错误响应：
- 项目组名称已存在 → `409 Conflict`，`message: "项目组名称已存在"`
- 参数校验失败 → `400 Bad Request`

后端逻辑：
- 校验项目组名称唯一（`uk_wb_project_group_name` 约束）
- `createdBy` 设为当前登录用户 ID
- `updatedBy` 设为当前登录用户 ID

### 7.3 删除项目组

```
DELETE /api/v1/groups/{id}
```

成功响应：`200 OK`。

```json
{
  "code": 200,
  "message": "success",
  "data": null
}
```

错误响应：
- 项目组不存在 → `404 Not Found`
- 项目组有成员 → `400 Bad Request`，`message: "该项目组尚有成员，无法删除"`
- 项目组有数据源（V1可选）→ `400 Bad Request`，`message: "该项目组尚有关联数据源，无法删除"`

后端逻辑：
- 校验项目组存在
- 校验 `wb_project_group_member` 中无该组记录
- 物理删除 `wb_project_group` 记录

## 八、错误提示文案（中文）

| 场景 | 文案 |
|---|---|
| 项目组名称为空 | `请输入项目组名称` |
| 项目组名称格式错误 | `项目组名称只能包含字母、数字、下划线和短横线，长度 2-64` |
| 项目组名称已存在 | `该项目组名称已被使用` |
| 描述过长 | `描述不能超过 255 个字符` |
| 创建成功 | `项目组 {name} 创建成功` |
| 删除确认 | `确定要删除项目组 {name} 吗？` |
| 删除说明 | `删除后不可恢复。` |
| 删除成功 | `项目组 {name} 已删除` |
| 有成员不可删除 | `该项目组尚有成员，无法删除` |

## 九、实施建议

### 后端

1. 扩展现有 `GroupController`（`/api/v1/groups`），增加分页列表、创建、删除接口
2. 新建 `GroupService`（业务逻辑：校验名称唯一、成员数统计、删除前检查）
3. 新建请求/响应 DTO：`CreateGroupRequest`、`GroupDetailResponse`（含 `memberCount`）
4. 保留现有 `GroupSimpleResponse` 用于用户管理下拉选择

### 前端

1. 新建 `src/api/group.ts` — API 层
2. 新建 `src/views/groups/GroupList.tsx` — 列表页（参照 UserList / DataSourceList）
3. 新建 `src/views/groups/GroupTable.tsx` — 表格组件
4. 新建 `src/views/groups/GroupForm.tsx` — 新建项目组对话框
5. 修改 `Layout.tsx` — 将「用户管理」导航项改造为「系统管理」下拉菜单（详见「顶部导航改造交互设计文档」）
6. 修改 `router/index.tsx` — 添加 `/groups` 路由（仅 systemAdmin 守卫，不需要 RequireGroup）

### 建议开发顺序

1. 后端 API（分页列表、创建、删除 3 个接口）
2. 前端列表页 + 搜索 + 分页
3. 新建项目组对话框
4. 删除项目组
5. 导航改造（与「顶部导航改造」文档联动）
6. 联调 + 浏览器验证
