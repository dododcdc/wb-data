# 用户 / 项目组 / 角色 V1 表结构设计

更新时间：2026-04-01

本文档把 [用户-项目组-角色-V1讨论纪要.md](/Users/wenbin/Projects/wb-data/用户-项目组-角色-V1讨论纪要.md) 里的逻辑模型，进一步落到当前项目可执行的 MySQL 表结构设计上，并顺便评估现有表需要怎么改。

## 一、先看当前现状

当前后端真正落地到数据库的只有一张表：

- `datasource`

对应现状证据：

- 迁移脚本里只有 [migration.sql](/Users/wenbin/Projects/wb-data/wb-data-server/wb-data-backend/src/main/resources/db/migration.sql#L1) 这一张表
- 实体类只有 [DataSource.java](/Users/wenbin/Projects/wb-data/wb-data-server/wb-data-backend/src/main/java/com/wbdata/datasource/entity/DataSource.java#L15) 绑定到了 `datasource`
- 查询导出任务当前不是数据库表，而是内存 `Map + 临时文件`，见 [QueryExportServiceImpl.java](/Users/wenbin/Projects/wb-data/wb-data-server/wb-data-backend/src/main/java/com/wbdata/query/service/impl/QueryExportServiceImpl.java#L32)

所以这次表结构设计的结论很明确：

- 现有表需要改的，主要是把 `datasource` 统一迁到 `wb_datasource`
- 需要新增的，是用户、项目组、成员关系、用户偏好、导出任务这几张表
- 图钉、查询历史、离线开发相关表，当前可以先不落，避免过度设计

## 二、设计原则

### 1. ID 类型

建议本项目 **统一使用 `BIGINT AUTO_INCREMENT`**，不要在当前阶段引入 UUID。

原因：

- 现有 `datasource.id` 已经是 `BIGINT`
- 当前后端和 MyBatis Plus 用自增主键最省改动
- 你现在是 V1，先把模型跑通比先追求“分布式 ID 统一”更重要

### 2. 角色设计

- 系统层：`SYSTEM_ADMIN / USER`
- 项目组层：`GROUP_ADMIN / DEVELOPER`

### 3. 命名

建议 **所有业务表统一加 `wb_` 前缀**。

推荐命名：

- `wb_user`
- `wb_project_group`
- `wb_project_group_member`
- `wb_user_group_preference`
- `wb_query_export_task`
- `wb_datasource`

原因：

- 一眼能区分哪些表属于本项目业务
- 避免 `user` 这类通用名带来的歧义
- 后续加离线开发、任务运维相关表时命名风格一致
- 当前项目还处于设计/早期实现阶段，现在统一成本最低

### 4. 审计字段

管理型表统一带：

- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

个人记录表只带：

- `created_at`
- `updated_at`

不强行加 `created_by / updated_by`，因为操作者就是 `user_id`

## 三、建议新增的表

### 1. `wb_user`

用途：

- 平台登录账号
- 存系统层角色
- 存账号状态

建议结构：

```sql
CREATE TABLE `wb_user` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `username` VARCHAR(64) NOT NULL COMMENT '登录名，唯一',
    `password_hash` VARCHAR(255) NOT NULL COMMENT '登录密码哈希',
    `display_name` VARCHAR(64) NOT NULL COMMENT '展示名',
    `system_role` VARCHAR(32) NOT NULL DEFAULT 'USER' COMMENT 'SYSTEM_ADMIN | USER',
    `status` VARCHAR(32) NOT NULL DEFAULT 'ACTIVE' COMMENT 'ACTIVE | DISABLED',
    `last_login_at` DATETIME NULL COMMENT '最近登录时间',
    `created_by` BIGINT NULL COMMENT '创建人',
    `updated_by` BIGINT NULL COMMENT '更新人',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_wb_user_username` (`username`),
    KEY `idx_wb_user_status` (`status`),
    CONSTRAINT `fk_wb_user_created_by` FOREIGN KEY (`created_by`) REFERENCES `wb_user` (`id`) ON DELETE SET NULL,
    CONSTRAINT `fk_wb_user_updated_by` FOREIGN KEY (`updated_by`) REFERENCES `wb_user` (`id`) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统用户';
```

补充说明：

- `display_name` 先保留，后面前端展示用户名、昵称都会更灵活
- 首个 `SYSTEM_ADMIN` 的 `created_by` 允许为 `NULL`

### 2. `wb_project_group`

用途：

- 代表业务上的项目组 / 空间 / 团队

建议结构：

```sql
CREATE TABLE `wb_project_group` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `name` VARCHAR(100) NOT NULL COMMENT '项目组名称',
    `description` VARCHAR(255) NULL COMMENT '项目组描述',
    `created_by` BIGINT NOT NULL COMMENT '创建人',
    `updated_by` BIGINT NULL COMMENT '更新人',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_wb_project_group_name` (`name`),
    CONSTRAINT `fk_wb_project_group_created_by` FOREIGN KEY (`created_by`) REFERENCES `wb_user` (`id`),
    CONSTRAINT `fk_wb_project_group_updated_by` FOREIGN KEY (`updated_by`) REFERENCES `wb_user` (`id`) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='项目组';
```

### 3. `wb_project_group_member`

用途：

- 表示某个用户属于哪个项目组
- 同时记录该用户在该项目组下的角色

建议结构：

```sql
CREATE TABLE `wb_project_group_member` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `group_id` BIGINT NOT NULL COMMENT '项目组ID',
    `user_id` BIGINT NOT NULL COMMENT '用户ID',
    `role` VARCHAR(32) NOT NULL COMMENT 'GROUP_ADMIN | DEVELOPER',
    `created_by` BIGINT NOT NULL COMMENT '创建人',
    `updated_by` BIGINT NULL COMMENT '更新人',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_wb_group_user` (`group_id`, `user_id`),
    KEY `idx_wb_group_member_user` (`user_id`),
    KEY `idx_wb_group_member_group_role` (`group_id`, `role`),
    CONSTRAINT `fk_wb_group_member_group` FOREIGN KEY (`group_id`) REFERENCES `wb_project_group` (`id`) ON DELETE CASCADE,
    CONSTRAINT `fk_wb_group_member_user` FOREIGN KEY (`user_id`) REFERENCES `wb_user` (`id`) ON DELETE CASCADE,
    CONSTRAINT `fk_wb_group_member_created_by` FOREIGN KEY (`created_by`) REFERENCES `wb_user` (`id`),
    CONSTRAINT `fk_wb_group_member_updated_by` FOREIGN KEY (`updated_by`) REFERENCES `wb_user` (`id`) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='项目组成员';
```

补充说明：

- 一个用户在一个项目组内只能有一个角色，所以 `(group_id, user_id)` 唯一
- 后面查“某人有哪些组”或“某组有哪些人”都靠这张表

### 4. `wb_user_group_preference`

用途：

- 存用户在某个项目组下的个人偏好
- 当前 V1 最重要的是默认数据源和最近访问时间

建议结构：

```sql
CREATE TABLE `wb_user_group_preference` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `user_id` BIGINT NOT NULL COMMENT '用户ID',
    `group_id` BIGINT NOT NULL COMMENT '项目组ID',
    `default_datasource_id` BIGINT NULL COMMENT '默认数据源ID',
    `last_accessed_at` DATETIME NULL COMMENT '最近访问该项目组时间',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_wb_user_group_preference` (`user_id`, `group_id`),
    CONSTRAINT `fk_wb_ugp_user` FOREIGN KEY (`user_id`) REFERENCES `wb_user` (`id`) ON DELETE CASCADE,
    CONSTRAINT `fk_wb_ugp_group` FOREIGN KEY (`group_id`) REFERENCES `wb_project_group` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户在项目组下的个人偏好';
```

补充说明：

- `default_datasource_id` 可以先不加外键，等 `wb_datasource` 改完再补
- 这是个人偏好表，不需要 `created_by / updated_by`

### 5. `wb_query_export_task`

用途：

- 把当前“内存里的导出任务”改成持久化
- 满足“导出任务属于用户 + 项目组上下文”的模型

建议结构：

```sql
CREATE TABLE `wb_query_export_task` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `task_uuid` CHAR(36) NOT NULL COMMENT '对外暴露的任务ID',
    `user_id` BIGINT NOT NULL COMMENT '发起导出的用户',
    `group_id` BIGINT NOT NULL COMMENT '导出所属项目组',
    `datasource_id` BIGINT NOT NULL COMMENT '数据源ID',
    `database_name` VARCHAR(128) NULL COMMENT '执行时所选数据库',
    `sql_text` LONGTEXT NOT NULL COMMENT '导出时提交的SQL快照',
    `format` VARCHAR(16) NOT NULL COMMENT 'csv | xlsx',
    `status` VARCHAR(32) NOT NULL COMMENT 'PENDING | RUNNING | SUCCESS | FAILED',
    `file_name` VARCHAR(255) NULL COMMENT '下载文件名',
    `file_path` VARCHAR(500) NULL COMMENT '文件存储路径',
    `row_limit` INT NOT NULL DEFAULT 100000 COMMENT '导出行数上限',
    `exported_rows` INT NULL COMMENT '实际导出行数',
    `truncated` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否截断',
    `error_message` VARCHAR(500) NULL COMMENT '失败原因',
    `expires_at` DATETIME NULL COMMENT '文件过期时间',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_wb_query_export_task_uuid` (`task_uuid`),
    KEY `idx_wb_query_export_task_user_group_created` (`user_id`, `group_id`, `created_at`),
    KEY `idx_wb_query_export_task_status` (`status`),
    CONSTRAINT `fk_wb_qet_user` FOREIGN KEY (`user_id`) REFERENCES `wb_user` (`id`) ON DELETE CASCADE,
    CONSTRAINT `fk_wb_qet_group` FOREIGN KEY (`group_id`) REFERENCES `wb_project_group` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='查询导出任务';
```

补充说明：

- 当前导出接口已经是异步任务模型，所以这张表是自然落地点
- `task_uuid` 保留字符串 ID，兼容现有 API 语义
- `sql_text` 建议存，方便排查问题

## 四、现有表 `datasource` 需要怎么改

当前表结构见 [migration.sql](/Users/wenbin/Projects/wb-data/wb-data-server/wb-data-backend/src/main/resources/db/migration.sql#L1)。

### 必改项

#### 0. 先统一表名

既然本项目决定采用 `wb_` 前缀方案，建议先把现有表重命名：

```sql
RENAME TABLE `datasource` TO `wb_datasource`;
```

原因：

- 否则后面会出现一组新表都带 `wb_`，唯独数据源表不带，风格会永久割裂
- 当前项目数据库体量还小，重命名成本可控
- 现在统一比后面接口、SQL、实体越来越多时再改更省事

#### 1. 增加项目组归属

```sql
ALTER TABLE `wb_datasource`
    ADD COLUMN `group_id` BIGINT NULL COMMENT '所属项目组ID' AFTER `id`;
```

原因：

- 数据源在 V1 里已经明确是“项目组共享资源”
- 后端后续所有数据源列表/详情/写操作都必须带项目组上下文

#### 2. 增加审计字段

```sql
ALTER TABLE `wb_datasource`
    ADD COLUMN `created_by` BIGINT NULL COMMENT '创建人ID' AFTER `owner`,
    ADD COLUMN `updated_by` BIGINT NULL COMMENT '更新人ID' AFTER `created_by`;
```

原因：

- `wb_datasource` 属于管理型资源
- 需要和新设计保持统一

#### 3. 增加索引和唯一约束

```sql
ALTER TABLE `wb_datasource`
    ADD UNIQUE KEY `uk_wb_datasource_group_name` (`group_id`, `name`),
    ADD KEY `idx_wb_datasource_group_status_created` (`group_id`, `status`, `created_at`),
    ADD KEY `idx_wb_datasource_group_type` (`group_id`, `type`);
```

原因：

- 数据源名称不一定要全局唯一，但建议“组内唯一”
- 数据源列表页未来一定按 `group_id` 查询

### 建议废弃项

#### `owner`

当前表里有：

```sql
`owner` VARCHAR(50) COMMENT '创建者'
```

这个字段建议进入废弃流程：

- 短期兼容阶段先保留，避免一下子改爆现有代码
- 等后端接口和实体改成 `created_by / updated_by` 后，再删除 `owner`

原因：

- `owner` 是字符串，不是可靠外键
- 和新的审计模型重复
- 也不适合多用户、多项目组场景

### 最终目标结构

`wb_datasource` 最终建议至少具备这些字段：

- `id`
- `group_id`
- `name`
- `type`
- `description`
- `host`
- `port`
- `database_name`
- `username`
- `password`
- `connection_params`
- `status`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

## 五、当前不建议现在就建的表

### 1. 图钉表

图钉在文档里有概念，但当前代码里还没有完整功能。建议等查询页面的 pin 功能真正开始实现时再建。

建议未来表名：

- `wb_query_pin`

### 2. 查询历史表

文档已经明确 V1 不做查询历史，所以现在不要建。若后续需要，建议表名统一为：

- `wb_query_history`

### 3. 离线开发相关表

当前离线开发还没有最终定是完全自研存储，还是更多依赖 Kestra / SeaTunnel / 外部编排系统。这个阶段先不要急着建 `wb_offline_task / wb_offline_node / wb_offline_run` 之类的表，否则大概率返工。

## 六、推荐实施顺序

建议按下面顺序做，不容易打架：

1. 先建 `wb_user`
2. 再建 `wb_project_group`
3. 再建 `wb_project_group_member`
4. 把 `datasource` 重命名并改造成 `wb_datasource`
5. 再建 `wb_user_group_preference`
6. 最后把导出任务从内存迁到 `wb_query_export_task`

## 七、总结

这轮设计的核心结论是：

- **新增 5 张表**：`wb_user`、`wb_project_group`、`wb_project_group_member`、`wb_user_group_preference`、`wb_query_export_task`
- **必须改 1 张现有表**：把 `datasource` 统一迁到 `wb_datasource`
- **当前不用急着建的表**：`wb_query_pin`、`wb_query_history`、离线开发相关表

如果你认可这版结构，下一步我建议直接做两件事中的一件：

1. 把这份设计转成真正的迁移 SQL 草稿文件
2. 继续往下推，先把后端 Java 实体和 mapper 也一起按这版结构起出来
