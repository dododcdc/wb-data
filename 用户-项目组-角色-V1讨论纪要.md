# 用户 / 项目组 / 角色 V1 讨论纪要

更新时间：2026-04-01

本文档用于记录当前已经讨论过的方案、已经确定的原则、推荐的 V1 方向，以及下次需要继续深入讨论的问题。

## 一、这次讨论的背景

当前在考虑系统下一阶段的方向时，出现了三个候选项：

1. 继续打磨"数据源管理"和"自助查询"
2. 引入"用户"，评估引入用户后的影响点
3. 进入初步设想中的"离线开发"模块

当前建议优先级：

1. 先做"用户 / 项目组 / 角色"的 V1 设计
2. 再继续打磨"数据源管理"和"自助查询"
3. 最后再进入"离线开发"模块

原因：

- "用户"是横切能力，会影响数据源、查询、导出任务、默认配置、后续离线任务的归属和权限。
- 如果先做离线开发，后续很可能因为权限和归属模型不清晰而返工。
- 如果继续深度打磨现有功能，也可能把很多单用户假设做得更深，后续更难调整。

## 二、当前推荐的 V1 总体方案

当前推荐采用最小可落地方案：

- 系统层角色只有两种：
  - `SYSTEM_ADMIN`
  - `USER`
- 项目组层角色先收敛到两种，不做自定义：
  - `GROUP_ADMIN`
  - `DEVELOPER`
- 用户可以属于多个项目组
- 同一个用户在不同项目组里可以有不同角色
- 系统管理员和项目组管理员是两回事
- 核心资源优先按"项目组"归属，而不是按"个人"归属
- 权限是统一来源，前后端共用一套权限定义
- V1 不做公开注册
- V1 不做审批流
- V1 不做自定义角色
- V1 不做"后端动态返回菜单树"

## 三、几个核心概念的解释

### 1. 为什么分 `SYSTEM_ADMIN` 和 `USER`

这是系统层角色，不是项目组内角色。

- `SYSTEM_ADMIN` 管整个平台
- `USER` 是普通账号

`SYSTEM_ADMIN` 负责的事情包括：

- 创建项目组
- 创建用户或邀请用户
- 将用户加入项目组
- 指定项目组管理员
- 查看所有项目组资源
- 处理平台级配置

普通 `USER` 不天然拥有这些能力。普通用户能做什么，要看他在某个项目组里的角色。

所以这里有两层：

- 系统层：`SYSTEM_ADMIN / USER`
- 项目组层：`GROUP_ADMIN / DEVELOPER`

### 2. "资源归属优先按项目组，不是按个人"是什么意思

意思是：系统里的核心资源默认属于某个项目组，而不是属于某个个人账号。

但不是所有资源都归项目组。资源分两类：

**共享资源（归属项目组）** — 团队需要协作使用的实体，人员变动后资源必须留下来：

- 数据源
- 后续离线任务、脚本、产物

**个人记录（归属用户 + 项目组上下文）** — 个人使用痕迹，不需要别人看到，但需要知道"是在哪个项目组下产生的"：

- 导出任务
- 图钉（pin）
- 默认数据源偏好
- 查询历史（未来若做）

"用户 + 项目组上下文"的意思是：数据存在 `user_id + group_id` 的维度下。用户切换项目组时，看到的是自己在该项目组下的个人记录。

这样的好处：

- 不会把个人操作痕迹暴露给同事
- 用户退出项目组后，个人记录可以失去可见性，不影响团队资产
- 数据源等真正的共享资源仍然留在项目组里

### 3. "权限驱动菜单，不是菜单驱动权限"是什么意思

正确顺序是：

1. 后端先判断当前用户有什么 `permission`
2. 前端根据这些 `permission` 决定显示哪些菜单和按钮
3. 后端接口再根据这些 `permission` 做真正的访问控制

这里的关键是：

- 权限是根
- 菜单只是表现层

不要反过来做成：

- 因为前端显示了菜单，所以默认用户就有权限
- 或者先配置菜单，再去猜权限

UI 不能代替安全控制。真正的权限拦截必须在后端。

### 4. `permission` 是什么

`permission` 是最细粒度的能力标识。

它的核心用途：

- 控制后端接口访问权限

它的附加用途：

- 驱动前端菜单、页面、按钮的显示与隐藏

### 5. "查询"相关概念辨析

本文档中"查询"一词在不同上下文中指代不同概念，需要区分：

| 概念 | 含义 | 对应 permission | 归属 |
|---|---|---|---|
| **查询能力** | 在自助查询页面执行 SQL 的权限 | `query.use` | 由角色权限控制 |
| **查询结果导出** | 把查询结果导出为文件的操作，会生成一条导出任务记录 | `query.export` | 导出任务归个人 + 项目组上下文 |
| **查询历史** | 用户执行过的 SQL 记录（V1 未实现，未来若做） | 待定 | 归个人 + 项目组上下文 |

注意：

- "查询能力"是权限控制问题，不涉及数据归属
- "查询结果导出"产生的导出任务是个人记录，管理员不可见
- "查询历史"当前系统不存在此功能，V1 不做，后续若做归个人

## 四、已确定的原则

以下原则已经明确确认：

1. 用户可以属于多个项目组
2. 同一个用户在不同项目组下可以有不同角色
3. 系统管理员和项目组管理员是两回事
4. 核心资源归属按两类划分：共享资源归项目组，个人记录归用户 + 项目组上下文
5. 权限是统一来源，前后端共用一套权限定义
6. 所有表统一带 `created_at` + `updated_at`
7. 管理操作表加 `created_by` / `updated_by`，个人操作表不加

## 五、V1 运作方式（已确认）

### 1. 用户创建方式

- V1 不开放公开注册
- 用户由 `SYSTEM_ADMIN` 在后台直接创建（用户名 + 初始密码）
- 管理员将凭据告知用户
不做：

- 公开注册
- 邀请链接激活
- 强制首次登录改密（V1 用户量小，口头提醒即可，后续可补）

### 首个 SYSTEM_ADMIN 的初始化

系统首次部署时，通过环境变量指定首个管理员账号：

```bash
INIT_ADMIN_USERNAME=admin
INIT_ADMIN_PASSWORD=your-chosen-password
```

启动时自动检查：如果 User 表为空，读取环境变量创建首个 SYSTEM_ADMIN（密码 hash 后写入）；如果已存在用户，跳过。不通过 Web 界面创建首个管理员。

### 2. 用户首次进入系统

**普通用户（system_role = USER）**，如果不属于任何项目组：

- 隐藏所有业务菜单（数据源、查询、离线开发、成员管理都不显示）
- 只显示空状态页：标题"你还未加入任何项目组"，描述"请联系系统管理员将你加入项目组后即可开始使用"
- 不显示"加入项目组"按钮或邀请码输入框（V1 不做）
- 顶部导航栏保留用户头像/退出入口，项目组选择器显示为空

**SYSTEM_ADMIN 不受此规则约束。** SYSTEM_ADMIN 即使不属于任何项目组，登录后也应直接进入系统管理界面（用户管理、项目组管理等），而不是空状态页。SYSTEM_ADMIN 的首页入口是系统管理，不是项目组工作台。

### 3. 用户加入项目组

- 由 `SYSTEM_ADMIN` 或 `GROUP_ADMIN` 将用户拉入项目组
- 用户加入项目组后，由管理员给他分配项目组内角色

V1 不做：

- 申请加入项目组
- 审批流
- 多级审核

### 4. 项目组切换方式

- 顶部导航栏放一个项目组选择器（下拉或弹窗），切换当前项目组
- 登录后默认进入上次所在的项目组
- 如果上次的项目组已被移除，则进入用户所属的第一个项目组
- 如果用户不属于任何项目组，进入空状态页

切换项目组后：

- 前端重新拉取当前项目组的权限列表
- 刷新数据源列表、导出任务等关联数据
- URL 中携带项目组标识（如 `/groups/{groupId}/datasources`），链接可分享

### 5. 角色来源

- V1 的项目组角色固定
- 不允许项目组管理员自定义角色
- 不做角色模板

## 六、角色结构（已确认）

### 系统层角色

- `SYSTEM_ADMIN` — 平台管理员，自动拥有所有项目组的全部权限，不需要单独加入每个组
- `USER` — 普通用户，能做什么取决于所属项目组内的角色

### 项目组层角色

- `GROUP_ADMIN` — 项目组管理者，组内全部权限
- `DEVELOPER` — 开发者，能查看数据源详情、执行查询与导出、做离线开发，但不能管理成员、项目组设置和数据源写操作

### SYSTEM_ADMIN 的特殊能力

| 能力 | 说明 |
|---|---|
| 创建用户 | 后台创建新账号 |
| 禁用/启用用户 | 冻结账号 |
| 创建项目组 | 新建项目组 |
| 删除项目组 | V1 只允许删除无成员、无数据源的空项目组，不做强制删除 |
| 将用户加入任何项目组 | 不限于自己管理的组 |
| 查看所有项目组的资源 | 全局可见性 |
| 管理平台级配置 | 如果有的话 |

SYSTEM_ADMIN 拥有超越项目组角色的系统级权限，可绕过组内权限检查，直接访问任意项目组的资源和操作。注意：这不是说 SYSTEM_ADMIN "等同于" GROUP_ADMIN —— 系统管理员和项目组管理员仍然是两个不同层级的概念，只是 SYSTEM_ADMIN 在权限判断上不受项目组角色约束。

**SYSTEM_ADMIN 可以切换到任意项目组上下文进行查看和操作。** 前端实现上，SYSTEM_ADMIN 的项目组选择器显示系统中所有项目组（不限于自己加入的）。SYSTEM_ADMIN 默认进入系统管理界面；当主动切换到某个项目组时，再进入该组的工作台，并拥有该组内全部权限。

### GROUP_ADMIN 的能力边界

- ✅ 可以从已有用户中添加成员到本组
- ✅ 可以把组内成员提升为 GROUP_ADMIN
- ✅ 可以创建/编辑/删除本组数据源（仅 GROUP_ADMIN 拥有数据源写权限）
- ✅ 可以查看本组所有离线任务和脚本（共享资源）
- ❌ 不可以查看其他成员的个人记录（导出任务、图钉、查询历史等属于个人记录，不对管理员开放）
- ❌ 不可以创建新用户（这是 SYSTEM_ADMIN 的事）
- ❌ 最后一个 GROUP_ADMIN 不可自行降级或退出项目组，需由 SYSTEM_ADMIN 操作（项目组至少保留一个 GROUP_ADMIN）

## 七、权限模型（已确认）

### V1 Permission 清单

| Permission | 含义 | 控制什么 |
|---|---|---|
| `datasource.read` | 查看数据源列表和详情 | 数据源列表页、详情页 |
| `datasource.write` | 创建/编辑/删除数据源 | 数据源管理操作 |
| `query.use` | 使用自助查询 | 执行 SQL 查询 |
| `query.export` | 导出查询结果 | 导出按钮/接口 |
| `offline.read` | 查看离线任务/脚本 | 离线开发模块只读 |
| `offline.write` | 创建/编辑/运行离线任务 | 离线开发模块写操作 |
| `member.read` | 查看项目组成员列表 | 成员列表页 |
| `member.manage` | 添加/移除成员、变更角色 | 成员管理操作 |
| `group.settings` | 修改项目组设置 | 项目组名称、描述等 |

### 角色-权限映射

| Permission | GROUP_ADMIN | DEVELOPER |
|---|---|---|
| `datasource.read` | ✅ | ✅ |
| `datasource.write` | ✅ | ❌ |
| `query.use` | ✅ | ✅ |
| `query.export` | ✅ | ✅ |
| `offline.read` | ✅ | ✅ |
| `offline.write` | ✅ | ✅ |
| `member.read` | ✅ | ✅ |
| `member.manage` | ✅ | ❌ |
| `group.settings` | ✅ | ❌ |

**SYSTEM_ADMIN 与权限矩阵的关系：**

- 上述权限矩阵仅适用于项目组层角色：`GROUP_ADMIN / DEVELOPER`
- `SYSTEM_ADMIN` 不参与项目组角色矩阵
- `SYSTEM_ADMIN` 默认拥有所有项目组的全部 permission，并可绕过组内权限检查
- 在前端展示上，`SYSTEM_ADMIN` 进入某个项目组上下文后，视为拥有该组内全部功能入口和操作权限

### 权限与菜单的关系

- 后端返回当前用户在当前项目组下的权限列表
- 前端本地维护菜单配置
- 前端根据权限列表过滤菜单和按钮显示

不做 V1：

- 后端直接返回整棵菜单树给前端初始化

## 八、资源归属模型（已确认）

### 共享资源（归属项目组）

| 资源 | 归属 | 理由 |
|---|---|---|
| 数据源 | 项目组 | 团队共用基础设施 |
| 离线任务/脚本/产物 | 项目组 | 团队资产，需要协作和交接 |

### 个人记录（归属用户 + 项目组上下文）

| 资源 | 归属 | 理由 |
|---|---|---|
| 导出任务 | 用户 + 项目组上下文 | 个人发起的操作 |
| 图钉（pin） | 用户 + 项目组上下文 | 个人操作 |
| 默认数据源偏好 | 用户 + 项目组上下文 | 个人偏好 |
| 查询历史（未来若做） | 用户 + 项目组上下文 | 个人行为记录 |

### 用户被移出项目组后的个人记录处理原则

当用户被移出某个项目组后：

- 该用户不再能访问该项目组上下文下的任何内容
- 该用户在该项目组下产生的个人记录（如导出任务、图钉、未来可能的查询历史）默认保留，不立即物理删除
- 这些个人记录不再对该用户可见，也不对项目组管理员开放
- 保留这些记录的目的，是避免破坏历史追踪和后续审计能力

V1 先采用“逻辑上不可见、数据上保留”的原则，不做自动清理策略

## 九、数据模型（已确认）

以下为逻辑模型。具体到当前项目的 MySQL 表结构落地设计，见：

- [用户-项目组-角色-V1表结构设计.md](/Users/wenbin/Projects/wb-data/用户-项目组-角色-V1表结构设计.md)

### User

```
User {
  id: UUID
  username: string (唯一)
  password_hash: string
  display_name: string
  system_role: SYSTEM_ADMIN | USER
  status: ACTIVE | DISABLED
  created_by: UUID -> User (nullable, 首个 admin 为 null)
  updated_by: UUID -> User (nullable)
  created_at: timestamp
  updated_at: timestamp
}
```

### User.status 的规则

- `ACTIVE`：可正常登录和使用系统
- `DISABLED`：不可登录
- 如果用户被设为 `DISABLED` 时系统中仍存在有效会话，则该会话应视为失效
- `DISABLED` 用户不可访问任何系统内容，包括系统管理界面和项目组工作台

### ProjectGroup

```
ProjectGroup {
  id: UUID
  name: string (唯一)
  description: string
  created_by: UUID -> User
  updated_by: UUID -> User (nullable)
  created_at: timestamp
  updated_at: timestamp
}
```

### Membership

```
Membership {
  id: UUID
  user_id: UUID -> User
  group_id: UUID -> ProjectGroup
  role: GROUP_ADMIN | DEVELOPER
  created_by: UUID -> User
  updated_by: UUID -> User (nullable)
  created_at: timestamp
  updated_at: timestamp

  UNIQUE(user_id, group_id)  // 同一用户在同一组只有一个角色
}
```

### UserGroupPreference

```
UserGroupPreference {
  id: UUID
  user_id: UUID -> User
  group_id: UUID -> ProjectGroup
  default_datasource_id: UUID -> DataSource (nullable)
  last_accessed_at: timestamp
  created_at: timestamp
  updated_at: timestamp

  UNIQUE(user_id, group_id)
}
```

### 现有表的改动

- **数据源表**：加 `group_id`（共享资源，归属项目组）+ `created_by` / `updated_by`
- **导出任务表**：加 `user_id` + `group_id`（个人记录 + 项目组上下文），不加 `created_by` / `updated_by`
- **图钉表**：加 `user_id` + `group_id`（个人记录 + 项目组上下文），不加 `created_by` / `updated_by`

### 字段策略

- 所有表统一带 `created_at` + `updated_at`
- 管理操作表（User、ProjectGroup、Membership、DataSource 等）加 `created_by` / `updated_by`
- 个人操作表（导出任务、图钉、偏好等）不加 `created_by` / `updated_by`，因为操作者就是 `user_id` 本人
