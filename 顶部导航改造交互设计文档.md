# 顶部导航改造交互设计文档

更新时间：2026-04-05

本文档定义顶部导航栏改造的前后端交互规范，供开发实施参考。

## 一、改造定位

当前顶部导航栏的「用户管理」是一个直接链接菜单项。本次改造需要：

1. **新增「系统管理」下拉菜单**（替代原来的「用户管理」直接链接），包含「用户管理」和「项目组」两个子项
2. **新增「成员与设置」导航入口**，所有项目组成员可见
3. 调整导航项的排列顺序和权限控制

## 二、改造前后对比

### 改造前（当前状态）

```
普通用户：    首页 | 数据源管理 | 自助查询
SYSTEM_ADMIN：首页 | 数据源管理 | 自助查询 | 用户管理
```

- 「用户管理」是一级菜单，直接链接到 `/users`
- 仅 `systemAdmin` 可见

### 改造后（目标状态）

```
普通用户：    首页 | 数据源管理 | 自助查询 | 成员与设置
SYSTEM_ADMIN：首页 | 数据源管理 | 自助查询 | 成员与设置 | 系统管理 ▾
                                                         ├ 用户管理
                                                         └ 项目组
```

- 「成员与设置」是一级菜单，链接到 `/group-settings`，所有已加入项目组的用户可见
- 「系统管理」是下拉菜单，仅 `systemAdmin` 可见，包含两个子项
- 原来的「用户管理」从一级菜单降为「系统管理」的子菜单项

## 三、导航项定义

### 3.1 一级菜单项

| 菜单项 | 路由 | 图标 | 可见条件 | 说明 |
|---|---|---|---|---|
| 首页 | `/` | `Home` | 始终可见 | 仪表盘 |
| 数据源管理 | `/datasources` | `Database` | `datasource.read` 权限 | 需要 RequireGroup |
| 自助查询 | `/query` | `Search` | `query.use` 权限 | 需要 RequireGroup |
| 成员与设置 | `/group-settings` | `Settings` | `member.read` 权限 | 需要 RequireGroup |
| 系统管理 | — | `Shield` | `systemAdmin === true` | 下拉菜单，无自身路由 |

### 3.2 「系统管理」下拉子菜单

| 子菜单项 | 路由 | 图标 | 说明 |
|---|---|---|---|
| 用户管理 | `/users` | `Users` | 原有功能，路由不变 |
| 项目组 | `/groups` | `FolderOpen` | 新增功能 |

## 四、「系统管理」下拉菜单交互规范

### 触发方式

- **点击触发**（非 hover）— 点击「系统管理」按钮展开下拉菜单
- 再次点击或点击菜单外部区域关闭

### 下拉菜单结构

```
┌────────────┐
│ 👤 用户管理 │
│ 📁 项目组   │
└────────────┘
```

### 视觉状态

- **默认态**：「系统管理」按钮样式与其他一级菜单一致，右侧有一个小三角 `▾` 表示可展开
- **展开态**：按钮高亮（同 active 状态），下方弹出下拉面板
- **子项 hover**：与普通菜单 hover 效果一致
- **子项 active**：当 URL 匹配 `/users` 或 `/groups` 时，「系统管理」按钮显示为 active，对应子项也高亮

### active 状态规则

| 当前 URL | 「系统管理」按钮状态 | 子菜单中高亮项 |
|---|---|---|
| `/users` | active | 用户管理 |
| `/groups` | active | 项目组 |
| 其他页面 | 默认 | 无 |

即：当 URL 属于系统管理任一子页面时，父级「系统管理」按钮应呈现 active 样式。

### 键盘可访问性

- `Tab` 可聚焦到「系统管理」按钮
- `Enter` / `Space` 展开下拉
- `↑` / `↓` 在子项间导航
- `Enter` 选择子项并导航
- `Escape` 关闭下拉

## 五、路由改造

### 新增路由

```
/group-settings  →  成员与设置页面（RequireGroup + RequirePermission("member.read")）
/groups          →  项目组管理页面（RequireSystemAdmin）
```

### 现有路由（不变）

```
/               →  首页
/datasources    →  数据源管理（RequireGroup + RequirePermission("datasource.read")）
/query          →  自助查询（RequireGroup + RequirePermission("query.use")）
/users          →  用户管理（RequireSystemAdmin）
```

### 路由守卫结构

```
AuthGuard
  └── Layout
      ├── / (Dashboard)
      ├── RequireGroup
      │   ├── /datasources (RequirePermission: datasource.read)
      │   ├── /query (RequirePermission: query.use)
      │   └── /group-settings (RequirePermission: member.read)
      ├── RequireSystemAdmin
      │   ├── /users (UserList)
      │   └── /groups (GroupList)
      └── * (NotFound)
```

## 六、代码改造要点

### 6.1 Layout.tsx 改造

#### navItems 改造

当前 `navItems` 数组中只有简单的一级链接。需要扩展数据结构以支持下拉菜单：

```typescript
interface NavItem {
    path: string;
    label: string;
    icon: LucideIcon;
    end?: boolean;
}

// 新增：下拉菜单型导航项
interface NavDropdownItem {
    label: string;
    icon: LucideIcon;
    children: NavItem[];
}

type NavigationItem = NavItem | NavDropdownItem;
```

#### 菜单构建逻辑

```typescript
const navItems = useMemo(() => {
    const hasPermission = (perm: string) => systemAdmin || permissions.includes(perm);
    const items: NavigationItem[] = [
        { path: '/', label: '首页', icon: Home, end: true },
    ];

    if (hasPermission('datasource.read')) {
        items.push({ path: '/datasources', label: '数据源管理', icon: Database });
    }

    if (hasPermission('query.use')) {
        items.push({ path: '/query', label: '自助查询', icon: Search });
    }

    if (hasPermission('member.read')) {
        items.push({ path: '/group-settings', label: '成员与设置', icon: Settings });
    }

    if (systemAdmin) {
        items.push({
            label: '系统管理',
            icon: Shield,
            children: [
                { path: '/users', label: '用户管理', icon: Users },
                { path: '/groups', label: '项目组', icon: FolderOpen },
            ],
        });
    }

    return items;
}, [permissions, systemAdmin]);
```

#### 下拉菜单渲染

在 `nav-menu` 中需要区分普通链接和下拉菜单：

- 普通链接：使用现有的 `<NavLink>` 组件
- 下拉菜单：使用独立的下拉组件（可复用 shadcn/ui 的 `DropdownMenu` 或自行实现）

### 6.2 router/index.tsx 改造

新增两个路由：

```typescript
// 在 RequireGroup 的 children 中添加：
{
    path: 'group-settings',
    element: <RequirePermission required="member.read" />,
    children: [
        {
            index: true,
            element: withRouteSuspense(<GroupSettings />),
        },
    ],
},

// 在 RequireSystemAdmin 的 children 中添加：
{
    path: 'groups',
    element: <RequireSystemAdmin />,
    children: [
        {
            index: true,
            element: withRouteSuspense(<GroupList />),
        },
    ],
},
```

注意：`/groups` 路由放在现有的 `RequireSystemAdmin` 下，与 `/users` 并列。

### 6.3 路由预加载（warmRoute）

扩展现有的 `warmRoute` 逻辑，新增 `/group-settings` 和 `/groups` 的模块预加载：

```typescript
if (path === '/group-settings') {
    void loadGroupSettingsModule();
}

if (path === '/groups') {
    void loadGroupListModule();
}
```

同时在 `routeModules.ts` 中新增对应的 lazy import 和 module loader。

## 七、实施建议

### 改造步骤

1. **扩展类型定义** — 新增 `NavDropdownItem` 类型
2. **实现下拉菜单组件** — 封装一个 `NavDropdown` 组件用于渲染「系统管理」菜单
3. **修改 Layout.tsx** — 更新 navItems 构建逻辑和渲染逻辑
4. **修改 router/index.tsx** — 添加 `/group-settings` 和 `/groups` 路由
5. **新增 routeModules** — 添加 `loadGroupSettingsModule` 和 `loadGroupListModule`
6. **样式调整** — 下拉菜单的 CSS 样式（与现有导航风格保持一致）

### 注意事项

- 下拉菜单的 active 状态检测：需要检查 `location.pathname` 是否匹配任一子项路由
- 路由预加载：hover 到「系统管理」时应预加载两个子页面模块
- 「成员与设置」依赖 `RequireGroup`，未加入项目组的用户（且非 SYSTEM_ADMIN）看不到此菜单项
- 移动端适配（V1 可暂不考虑，后续按需处理）

### 建议开发顺序

此文档应与其他两份文档联动开发：

1. 先完成后端 API（项目组管理 + 成员与设置）
2. 前端先完成两个新页面的基础框架
3. 然后做导航改造（本文档）
4. 最后联调验证所有导航路径
