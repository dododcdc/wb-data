# 成员与设置交互设计文档

更新时间：2026-04-05

本文档定义组级「成员与设置」页面的前后端交互规范，供开发实施参考。

## 一、模块定位

「成员与设置」是项目组级别功能，所有项目组成员（GROUP_ADMIN 和 DEVELOPER）均可进入。页面包含两部分：

1. **项目组信息卡片**（顶部）— 展示/编辑项目组名称和描述
2. **成员列表**（下方）— 查看/管理项目组成员

采用方案B设计：顶部卡片展示项目组信息 + 下方成员表格，不使用 Tab 或侧边栏。

### 角色能力概览

| 功能 | GROUP_ADMIN | DEVELOPER |
|---|---|---|
| 查看项目组信息 | ✅ | ✅ |
| 编辑项目组名称/描述 | ✅（`group.settings`） | ❌ |
| 查看成员列表 | ✅（`member.read`） | ✅（`member.read`） |
| 添加成员 | ✅（`member.manage`） | ❌ |
| 移除成员 | ✅（`member.manage`） | ❌ |
| 修改成员角色 | ✅（`member.manage`） | ❌ |

注：`SYSTEM_ADMIN` 默认拥有所有权限，等同于 GROUP_ADMIN。

## 三、导航入口

### 菜单位置

在顶部导航栏作为一级菜单项，所有已加入项目组的用户可见：

```
普通用户：    首页 | 数据源管理 | 自助查询 | 成员与设置
SYSTEM_ADMIN：首页 | 数据源管理 | 自助查询 | 成员与设置 | 系统管理 ▾
```

### 路由

- 路径：`/group-settings`
- 守卫：需要 `RequireGroup`（必须有当前项目组上下文）+ `RequirePermission("member.read")`
- 注意：此页面依赖项目组上下文 — 展示和操作的都是当前选中项目组的数据

## 五、项目组信息卡片

### 默认展示态

卡片展示项目组的名称和描述。

```
┌─────────────────────────────────────────────┐
│  项目组名称: alpha                    [编辑]  │
│  描述: Alpha 项目组                          │
└─────────────────────────────────────────────┘
```

- 项目组名称：粗体展示
- 描述：普通文本展示，无描述时显示灰色 `暂无描述`
- 「编辑」按钮：仅当用户拥有 `group.settings` 权限时显示（GROUP_ADMIN 或 SYSTEM_ADMIN）

### 编辑态

点击「编辑」按钮后，卡片切换为内联编辑模式：

```
┌─────────────────────────────────────────────┐
│  项目组名称 *                                │
│  [alpha_______________]                     │
│                                             │
│  描述                                       │
│  [Alpha 项目组________]                     │
│  [__________________]                       │
│                                             │
│                    [取消]  [保存]             │
└─────────────────────────────────────────────┘
```

### 字段规则

| 字段 | 必填 | 校验规则 | 说明 |
|---|---|---|---|
| 项目组名称 | ✅ | 2-64 字符，仅允许字母、数字、下划线、短横线；不可与其他项目组名重复 | 表有唯一约束 |
| 描述 | 否 | 最长 255 字符 | 多行文本输入（textarea） |

### 交互规则

1. 点击「编辑」→ 卡片切换为编辑模式，输入框回填当前值
2. 点击「取消」→ 放弃修改，恢复展示态
3. 点击「保存」→ 按钮进入 loading 态，显示 `保存中...`
4. 保存成功：
   - 卡片切换回展示态，显示新值
   - 提示 `项目组信息已更新`
   - **同步更新导航栏项目组选择器中的名称**（如果改了名称）
5. 保存失败：
   - 名称已存在 → 名称输入框下方显示红色提示：`该项目组名称已被使用`
   - 其他错误 → 卡片底部显示红色错误提示
   - 保持编辑态，不丢失输入内容

## 六、成员列表

### 表格列定义

| 列 | 字段 | 说明 |
|---|---|---|
| 用户名 | `username` | 成员的登录名 |
| 展示名 | `displayName` | 成员的显示名称 |
| 项目组角色 | `role` | 显示为标签：`项目组管理员` / `开发者` |
| 加入时间 | `createdAt` | 成员加入项目组的时间，格式 `YYYY-MM-DD HH:mm:ss` |
| 操作 | — | 操作按钮组（仅 GROUP_ADMIN 可见） |

### 搜索

- 列表标题行可选是否放搜索框
- 如果有搜索框：placeholder `搜索用户名或展示名`，防抖 300ms
- V1 项目组成员数量通常较少（几人到几十人），搜索为可选功能

### 分页

- 默认每页 10 条
- 可选：10 / 20 / 50
- 分页参数同步到 URL（`?page=1&size=10`）

### 操作按钮（仅 GROUP_ADMIN 或 SYSTEM_ADMIN 可见）

每行根据条件显示：

| 操作 | 图标 | 条件 | 说明 |
|---|---|---|---|
| 修改角色 | `UserCog` | 非当前登录用户 | 变更该成员的项目组角色 |
| 移除 | `UserMinus` | 非当前登录用户 且 非唯一 GROUP_ADMIN | 从项目组中移除该成员 |

**DEVELOPER 角色**的用户看不到操作列（或操作列为空）。

### 保护规则

- **不可修改自己的角色** — 当前登录用户的「修改角色」按钮不显示
- **不可移除自己** — 当前登录用户的「移除」按钮不显示
- **最后一个 GROUP_ADMIN 保护** — 如果组内只剩一个 GROUP_ADMIN，该成员的「移除」和「修改角色」按钮不显示。后端同样需校验

## 七、添加成员

### 入口

成员列表标题栏右侧「添加成员」按钮，仅拥有 `member.manage` 权限时显示。

### 对话框结构

```
┌──── 添加成员 ──────────────── [×] ─┐
│                                    │
│  选择用户 *                         │
│  [搜索用户名或展示名_______▾]       │
│                                    │
│  项目组角色 *                       │
│  [开发者 ▾]                        │
│                                    │
│           [取消]  [确认添加]         │
└────────────────────────────────────┘
```

### 字段规则

| 字段 | 必填 | 校验规则 | 说明 |
|---|---|---|---|
| 选择用户 | ✅ | 可搜索的下拉框，从系统用户列表中选择 | 仅显示尚未加入本项目组的 ACTIVE 用户 |
| 项目组角色 | ✅ | 下拉选择：`开发者`（默认）/ `项目组管理员` | — |

### 用户搜索下拉说明

- 下拉框支持输入搜索，防抖 300ms
- 后端接口返回不在当前项目组内的 ACTIVE 用户列表
- 每个选项展示：`用户名 — 展示名`
- 如果搜索无结果，显示 `未找到匹配的用户`

### 交互规则

1. 点击「确认添加」后按钮进入 loading 态
2. 添加成功：
   - 关闭对话框
   - 成员列表自动刷新
   - 提示 `已将 {displayName} 添加到项目组`
3. 添加失败：
   - 用户已在项目组中 → `该用户已是本项目组成员`
   - 用户不存在 → `指定的用户不存在`
   - 其他错误 → 对话框底部显示错误

## 八、修改成员角色

### 入口

成员列表操作列「修改角色」按钮。

### 对话框结构

```
┌──── 修改角色 ──────────────── [×] ─┐
│                                    │
│  修改 user1 的项目组角色             │
│                                    │
│  项目组角色                          │
│  [开发者 ▾]                        │
│                                    │
│           [取消]  [确认修改]         │
└────────────────────────────────────┘
```

### 交互规则

1. 打开时回填当前角色
2. 点击「确认修改」→ loading 态
3. 修改成功：
   - 关闭对话框
   - 成员列表刷新
   - 提示 `{displayName} 的角色已变更为 {新角色}`
4. 修改失败：
   - 是唯一 GROUP_ADMIN 降级 → `该成员是唯一的项目组管理员，无法降级`
   - 修改自己 → `不可修改自己的角色`
   - 其他错误 → 对话框内显示错误

## 九、移除成员

### 入口

成员列表操作列「移除」按钮。

### 交互规则

1. 点击「移除」按钮
2. 弹出确认对话框：
   ```
   确定要将 user1 从项目组中移除吗？
   移除后该用户将无法访问本项目组的资源。
   [取消]  [确认移除]
   ```
3. 「确认移除」按钮为红色/警告色
4. 确认后：
   - 成功 → 成员列表刷新 + 提示 `{displayName} 已从项目组中移除`
   - 失败 → 提示错误信息

### 保护规则（后端校验）

- 不可移除自己
- 不可移除组内唯一的 GROUP_ADMIN
- 后端需对上述规则做强校验，不依赖前端按钮隐藏

## 十、后端 API 设计

本模块涉及两类接口：项目组信息管理 和 成员管理。

### 权限说明

| 接口 | 权限要求 |
|---|---|
| 获取项目组信息 | `member.read`（已加入项目组的所有成员） |
| 更新项目组信息 | `group.settings`（GROUP_ADMIN 或 SYSTEM_ADMIN） |
| 成员列表 | `member.read` |
| 添加成员 | `member.manage`（GROUP_ADMIN 或 SYSTEM_ADMIN） |
| 修改成员角色 | `member.manage` |
| 移除成员 | `member.manage` |

所有接口均需 `Authorization: Bearer {token}` 且操作在当前项目组上下文中执行。

### 10.1 获取当前项目组信息

```
GET /api/v1/group-settings
```

通过当前认证上下文中的 `currentGroupId` 确定目标项目组。

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 1,
    "name": "alpha",
    "description": "Alpha 项目组",
    "createdAt": "2026-04-01T10:00:00"
  }
}
```

### 10.2 更新项目组信息

```
PUT /api/v1/group-settings
Content-Type: application/json

{
  "name": "alpha-v2",
  "description": "更新后的描述"
}
```

成功响应：`200 OK`，返回更新后的项目组对象。

错误响应：
- 项目组名称已存在 → `409 Conflict`，`message: "项目组名称已存在"`
- 无 `group.settings` 权限 → `403 Forbidden`
- 参数校验失败 → `400 Bad Request`

后端逻辑：
- 从当前认证上下文获取 `currentGroupId`
- 校验名称唯一（排除自身）
- `updatedBy` 设为当前登录用户 ID

### 10.3 成员分页列表

```
GET /api/v1/group-settings/members?page=1&size=10&keyword=user
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [
      {
        "id": 1,
        "userId": 2,
        "username": "user1",
        "displayName": "用户一",
        "role": "GROUP_ADMIN",
        "createdAt": "2026-04-01T10:00:00"
      }
    ],
    "total": 5,
    "size": 10,
    "current": 1,
    "pages": 1
  }
}
```

后端逻辑：
- 通过 `currentGroupId` 查询 `wb_project_group_member` 关联 `wb_user`
- `keyword` 模糊匹配 `username` 和 `display_name`
- 按 `role` 优先（GROUP_ADMIN 排前面），再按 `created_at` 排序

### 10.4 可添加用户列表（供下拉选择）

```
GET /api/v1/group-settings/available-users?keyword=user
```

响应：
```json
{
  "code": 200,
  "message": "success",
  "data": [
    { "id": 3, "username": "user3", "displayName": "用户三" },
    { "id": 4, "username": "user4", "displayName": "用户四" }
  ]
}
```

后端逻辑：
- 返回 `status = ACTIVE` 且不在当前项目组内的用户
- `keyword` 模糊匹配 `username` 和 `display_name`
- 不分页（通常结果有限）；如果系统用户很多，限制返回前 50 条

### 10.5 添加成员

```
POST /api/v1/group-settings/members
Content-Type: application/json

{
  "userId": 3,
  "role": "DEVELOPER"
}
```

成功响应：`201 Created`。

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 10,
    "userId": 3,
    "username": "user3",
    "displayName": "用户三",
    "role": "DEVELOPER",
    "createdAt": "2026-04-05T15:30:00"
  }
}
```

错误响应：
- 用户已是成员 → `409 Conflict`，`message: "该用户已是本项目组成员"`
- 用户不存在或已禁用 → `400 Bad Request`，`message: "指定的用户不存在或已禁用"`
- 无 `member.manage` 权限 → `403 Forbidden`

后端逻辑：
- 校验用户存在且 `status = ACTIVE`
- 校验用户不在当前项目组中
- 插入 `wb_project_group_member` 记录
- `createdBy` / `updatedBy` 设为当前登录用户 ID

### 10.6 修改成员角色

```
PUT /api/v1/group-settings/members/{memberId}/role
Content-Type: application/json

{
  "role": "GROUP_ADMIN"
}
```

成功响应：`200 OK`。

错误响应：
- 修改自己的角色 → `400 Bad Request`，`message: "不可修改自己的角色"`
- 唯一 GROUP_ADMIN 降级 → `400 Bad Request`，`message: "该成员是唯一的项目组管理员，无法降级"`
- 无 `member.manage` 权限 → `403 Forbidden`
- 成员记录不存在 → `404 Not Found`

后端逻辑：
- 校验目标成员 userId ≠ 当前登录用户 ID
- 如果是将 GROUP_ADMIN 降为 DEVELOPER：检查组内 GROUP_ADMIN 数量是否 > 1
- 更新 `role` 字段
- `updatedBy` 设为当前登录用户 ID

### 10.7 移除成员

```
DELETE /api/v1/group-settings/members/{memberId}
```

成功响应：`200 OK`。

错误响应：
- 移除自己 → `400 Bad Request`，`message: "不可移除自己"`
- 唯一 GROUP_ADMIN → `400 Bad Request`，`message: "该成员是唯一的项目组管理员，无法移除"`
- 无 `member.manage` 权限 → `403 Forbidden`
- 成员记录不存在 → `404 Not Found`

后端逻辑：
- 校验目标成员 userId ≠ 当前登录用户 ID
- 校验如果目标是 GROUP_ADMIN，组内 GROUP_ADMIN 数量是否 > 1
- 物理删除 `wb_project_group_member` 记录

## 十二、实施建议

### 后端

1. 新建 `GroupSettingsController`（`/api/v1/group-settings`）— 项目组信息和成员管理
