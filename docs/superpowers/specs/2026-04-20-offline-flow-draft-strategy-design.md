# 离线开发 Flow 草稿策略改造设计

**日期**：2026-04-20  
**状态**：已确认，待写实施计划  
**范围**：`wb-data-frontend` 离线开发工作台（Flow 编辑、节点编辑器、切换 Flow、刷新恢复）

---

## 1. 背景与问题

当前离线开发模块存在多层“暂存”语义：

1. 节点编辑器本地输入缓冲
2. 页面内存中的 `flowDocument`
3. 浏览器本地草稿（`localStorage`）
4. 本地仓库文件（点击保存后写入）

这套机制具备一定防丢稿能力，但产品语义不清晰，已经出现以下问题：

- 切换 Flow 时，未满 5 秒的修改可能丢失
- “应用暂存”和“自动草稿”是两套语义，用户难以判断当前修改是否已经被系统记住
- `dirty` 判定不完整，`dataSourceId / dataSourceType` 不参与签名比较
- 5 秒定时器依赖过宽，部分 UI 行为会重置保存时机
- 保存成功后仍残留本地草稿，后续恢复逻辑复杂
- 路径变化（如重命名 Flow）后，旧草稿缺少迁移/清理策略

本次改造目标不是继续修补 5 秒自动保存，而是把草稿模型收敛成**单一语义、用户可预测、实现边界清晰**的一套策略。

---

## 2. 设计目标

### 2.1 产品目标

1. 用户只需要理解两个概念：
   - **当前 Flow 草稿**
   - **保存到本地仓库**
2. 只要用户修改了当前 Flow，系统就应当记住这份草稿
3. 切换 Flow / 切项目组 / 刷新页面时，不应因为时间窗口导致改动丢失
4. “保存到本地仓库”必须保持为用户显式触发的动作，便于后续审计“谁点击了保存”

### 2.2 工程目标

1. 把草稿逻辑集中管理，避免继续堆叠在 `OfflineWorkbench.tsx`
2. 用事件驱动替代产品层“5 秒自动保存”语义
3. 把浏览器本地恢复快照与仓库保存彻底解耦
4. 为后续实现冲突处理、审计和更细粒度恢复留出稳定边界

---

## 3. 核心结论

### 3.1 保留“手动保存到仓库”，取消“5 秒自动保存”的产品语义

“保存 Flow”仍然是唯一会写入本地仓库文件的动作，并且必须由用户点击触发。

系统仍会自动做一件事，但不再称为“保存”：

> **自动本地恢复快照**

它只写浏览器本地（`localStorage` 或 IndexedDB），只用于异常恢复和切回继续编辑：

- 不写到服务器 repo 文件
- 不影响其他用户
- 不进入 commit / push
- 不记为正式保存行为

### 3.2 建立“单一草稿模型”

每个 Flow 在前端只有一个权威编辑态：

- `baseDocument`：当前已加载的仓库基线版本
- `workingDraft`：用户当前正在编辑的 Flow 草稿

用户所有修改都应进入 `workingDraft`。  
节点编辑器不再拥有业务语义上的“未应用内容”这一独立状态层，只保留必要的组件级输入缓冲。

---

## 4. 目标状态模型

### 4.1 `workingDraft`

`workingDraft` 代表当前 Flow 的真实编辑状态，至少包含：

- Flow 文档正文
- 节点脚本内容
- 节点数据源选择（`dataSourceId / dataSourceType`）
- 连线
- 布局
- 节点增删改名结果

可选恢复态：

- 当前选中节点
- 当前选中任务集合

### 4.2 `baseDocument`

`baseDocument` 用于表示当前仓库已知基线版本，至少包含：

- 服务端返回的 Flow 文档
- `documentHash`
- `documentUpdatedAt`

### 4.3 `dirty`

`dirty` 定义为：

> `workingDraft` 与 `baseDocument` 的完整业务字段比较结果

本次改造要求把以下字段统一纳入 dirty 判定：

- `taskId`
- `kind`
- `scriptPath`
- `scriptContent`
- `dataSourceId`
- `dataSourceType`
- `edges`
- `layout`
- 其他会影响保存结果的结构字段

不再允许出现“已改动但不算 dirty”的保存盲区。

---

## 5. 自动本地恢复快照

### 5.1 定义

自动本地恢复快照是 `workingDraft` 的浏览器本地持久化副本，目标是：

- 页面刷新后恢复
- 浏览器异常退出后恢复
- 切换到其他 Flow 后再切回恢复

### 5.2 存储结构

按 Flow 维度存储：

- key：`groupId + flowPath`
- value：
  - `workingDraft`
  - `baseVersionMeta`（如 `documentHash`、`documentUpdatedAt`）
  - `updatedAt`
  - 可选 UI 恢复态

### 5.3 产品语义

自动本地恢复快照不是“保存”。

UI 和文案必须清楚区分：

- **已更新当前草稿**：表示修改已进入 `workingDraft`
- **已保存到本地仓库**：表示点击保存成功，仓库文件已更新

禁止继续使用会混淆两者的文案，如“暂存成功”。

---

## 6. 触发规则

### 6.1 必触发：离开当前工作面前

以下动作发生前，必须先执行：

1. flush 当前节点编辑器输入到 `workingDraft`
2. 写入自动本地恢复快照
3. 再继续后续动作

适用场景：

- 切换到另一个 Flow 前
- 切换项目组前
- 页面路由离开前
- 浏览器刷新 / 关闭前
- 执行前
- 会重载当前 Flow 文件的动作前（如调度写回 Flow 文件）

### 6.2 补充触发：编辑完成点

以下动作发生后，应异步更新恢复快照：

- 删除节点
- 新增节点
- 节点改名
- 连线变化
- 布局变化
- 数据源选择变化
- 节点编辑器关闭
- 节点编辑器执行“关闭并保留到草稿”类动作

### 6.3 技术节流

允许对浏览器本地快照写入做轻量 debounce 或 idle 调度，但这只是性能优化，不构成产品语义。

换言之：

- **用户一修改，就已经算进入当前 Flow 草稿**
- debounce 只影响“何时把草稿写入浏览器本地”，不影响“是否已被系统记住”

因此，本次设计明确取消“5 秒自动保存”的产品层表达。

---

## 7. 节点编辑器策略

### 7.1 现状问题

当前节点编辑器存在两层容易混淆的内容：

- 编辑器内输入缓冲
- `flowDocument` 中的已应用内容

用户必须区分“我只是打字了”和“我点了应用暂存”，这会直接导致误判。

### 7.2 新策略

节点编辑器仍可保留局部输入 state 以保证交互流畅，但业务语义上：

- 输入内容属于当前 Flow 草稿的一部分
- 关闭编辑器前必须 flush 到 `workingDraft`
- 切换 Flow / 组 / 路由前必须 flush
- 执行前必须 flush

### 7.3 按钮语义调整

建议收敛现有按钮语义：

- 弱化或移除“应用暂存”
- 保留“保存 Flow”作为唯一正式保存动作
- 节点编辑器关闭时默认保留到草稿，不再要求用户理解双层暂存

可选呈现：

- **关闭并保留到草稿**
- **保存 Flow**

目标是让用户不再思考“我现在到底有没有真正暂存”。

---

## 8. 切换 Flow / 切组 / 刷新行为

### 8.1 统一离开流程

无论是切换 Flow、切组还是离开页面，都统一走一条离开流程：

1. `flushEditorBuffer()`
2. `persistRecoverySnapshot()`
3. 清理当前 Flow 的瞬时 UI 状态
4. 再执行切换或离开

### 8.2 解决的典型问题

改造完成后，以下场景必须成立：

1. 在 `totot` 删除节点
2. 立即切到 `tyy`
3. 再切回 `totot`

结果：

- 已删除节点不会因为“未满 5 秒”而回来
- 切换行为不再依赖时间窗口是否恰好触发

---

## 9. 恢复与冲突处理

### 9.1 打开 Flow 时的判断

打开某个 Flow 时：

1. 先读取仓库最新内容
2. 再读取该 Flow 的本地恢复快照
3. 根据快照和当前仓库内容的关系决定是否恢复或提示

### 9.2 用户可见文案原则

“快照基于旧版本”只是内部判断语，不直接暴露给用户。

用户只需要理解两个概念：

- **本地恢复稿**
- **仓库最新内容**

推荐提示文案：

> 发现未保存的本地恢复稿。当前文件也有更新。请选择继续恢复，或加载最新内容。

### 9.3 冲突场景

典型场景包括：

- 保存 Flow 时后端返回 409
- 另一个标签页或会话修改并保存了同一 Flow
- 调度等系统动作写回了当前 Flow 文件
- 浏览器里残留旧恢复稿，而仓库里的文件已经更新

### 9.4 用户选择

冲突时提供明确二选一：

- **继续恢复稿**：回到本机未正式保存的编辑状态
- **加载最新内容**：放弃当前本地恢复稿，使用仓库版本

不自动吞掉任一方结果。

## 10. 结构拆分建议

本次改造应将草稿逻辑从 `OfflineWorkbench.tsx` 中拆出，形成清晰边界。

### 10.1 `flowDraftController`

职责：

- 管理 `baseDocument`
- 管理 `workingDraft`
- 计算 `dirty`
- 管理本地恢复快照
- 处理保存到仓库
- 处理冲突状态

建议接口：

- `openFlow(path, serverDocument)`
- `patchDraft(patch | producer)`
- `flushEditorBuffer()`
- `persistRecoverySnapshot()`
- `saveToRepository()`
- `discardRecoverySnapshot()`
- `resolveConflict(strategy)`

### 10.2 `nodeEditorSession`

职责：

- 管理当前打开的节点编辑器
- 管理编辑器输入 state
- 在离开边界前把输入 flush 给 `flowDraftController`

它不再定义“是否已暂存”的业务语义。

### 10.3 `recoverySnapshotStore`

职责：

- 封装浏览器本地读写
- 定义 schema
- 管理 key 规则
- 处理清理与迁移
- 为重命名 / 删除 Flow 提供辅助迁移能力

---

## 11. 失败处理

### 11.1 本地恢复快照写失败

- 不阻断继续编辑
- 不丢失当前内存中的 `workingDraft`
- 提示一次轻量 warning：
  - 当前浏览器无法保存本地恢复稿，关闭页面后未保存修改可能丢失

### 11.2 保存 Flow 失败

- 保留 `workingDraft`
- 立即确保恢复快照可用
- 提示：
  - 未保存到本地仓库，但本机恢复稿已保留

### 11.3 409 冲突

- 进入“本地恢复稿 vs 仓库最新内容”的冲突态
- 用户显式选择恢复稿或最新内容

---

## 12. 渐进式改造策略

建议分两步执行：

### 第一步：统一语义

- 引入 `workingDraft`
- 把节点编辑器输入、数据源、画布修改统一并入 `workingDraft`
- 去掉 5 秒产品语义
- 改成事件驱动的本地恢复快照
- 切换 Flow / 组 / 路由前统一 flush + snapshot
- 补齐 dirty 比较字段

### 第二步：清理旧概念

- 弱化或移除“应用暂存”
- 统一成功/失败提示文案
- 清理旧 dead code 和漂移参数（如 `silentDraftRestore`）
- 为重命名 / 删除 Flow 增加恢复稿迁移或清理逻辑

---

## 13. 验收标准

改造完成后，以下场景必须成立：

1. 在 `totot` 删除节点后立即切到 `tyy` 再切回，修改仍然存在
2. 在节点编辑器输入 SQL，不点任何“应用”，直接切 Flow，再切回，输入仍能恢复
3. 只改数据源，不改 SQL，刷新后仍能恢复
4. 点击“保存 Flow”前，repo 文件不变；点击后 repo 文件才变化
5. 保存失败后，刷新页面仍能恢复当前草稿
6. 409 冲突时，用户能清楚理解“继续恢复稿”与“加载最新内容”的区别
7. 删除 / 重命名 Flow 后，不残留错绑的恢复稿

---

## 14. 非目标

本次改造不包含以下内容：

- 多人实时协同编辑
- 跨浏览器或跨设备同步草稿
- 自动提交到 Git
- 自动推送到远程仓库

这些能力若后续需要，应建立在本次单一草稿模型之上，而不是反向挤进当前改造范围。

---

## 15. 最终建议

本次改造应明确落地为：

1. **一个草稿语义**：所有用户修改都进入 `workingDraft`
2. **两个持久化层**：
   - 自动本地恢复快照（浏览器本地）
   - 用户点击保存后的仓库文件
3. **一个统一离开流程**：切换 Flow / 刷新 / 离开前统一 flush + snapshot

这能从根本上解决当前策略“复杂、难预测、存在时间窗口丢改动”的问题，并为后续审计“谁点击了保存”保留清晰边界。
