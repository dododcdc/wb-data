## 一、核心定位

离线开发模块是 wb-data 的任务编排平台，用于可视化编排和执行 Kestra Flow。
用户通过拖拽式 DAG 画布构建任务流程，文件存储于 GitHub（后期可切换 Gitea），执行引擎为 Kestra。

---

## 二、技术架构

| 组件 | 角色 |
|-----|------|
| GitHub | Git 远程仓库，**唯一真相源**，存储 Flow 定义和脚本文件（后期可切换 Gitea） |
| wb-data 后端服务器 | **本地 clone 一份仓库**，所有用户操作在本地进行 |
| Kestra | 执行引擎，从 GitHub SyncFlows 同步后执行任务 |
| wb-data 前端 | 用户界面 + 项目组管理 |

### 本地仓库策略

**wb-data 后端服务器 clone 一份仓库**：
- 所有用户通过 wb-data UI 操作本地仓库
- 操作完成后「提交」到本地，「推送」到 GitHub
- Kestra 从 GitHub 定时同步（SyncFlows）

**新建项目组时**：
1. GitHub 创建远程仓库
2. clone 到 wb-data 后端服务器（此时仓库为空，很快）

**仓库目录结构**：
```
wb-data 服务器上：
/data/repos/wb-data-{project_group_id}/
```

### GitHub 仓库策略

**一库一项目组**：每个项目组有独立 GitHub 仓库，仓库命名 `wb-data-{project_group_id}`，UI 显示项目组名称。

| 层级 | 命名方式 |
|-----|---------|
| namespace | `pg-{project_group_id}` |
| GitHub 仓库 | `wb-data-{project_group_id}` |
| wb-data 本地仓库 | `/data/repos/wb-data-{project_group_id}` |
| UI 显示 | 项目组名称（如"数据平台组"） |

**机器人账号**：所有代码操作通过系统级机器人账号（如 `wb-data-bot`）执行，不依赖个人账号。

**约束**：所有操作只能在 wb-data 界面进行，不允许去 GitHub Web 界面操作。

## 三、目录结构

使用 Kestra 官方约定的目录结构：

```
📁 GitHub 仓库
│
├── 📁 _flows/
│     └── 📁 data_pipeline/
│           └── 📁 etl/
│                 └── flow.yaml      ← namespace 写在文件里
│
└── 📁 scripts/
      └── 📁 data_pipeline/
            └── 📁 etl/
                  └── query.sql
```

**说明**：
- `_flows/` 是 Kestra SyncFlows 的扫描根目录（约定，不可更改）
- `scripts/` 存放 Flow 引用的脚本文件，可与 `_flows/` 下的任务目录结构对应
- namespace 定义在 `flow.yaml` 文件内部，与目录结构无关
- `_flows/` 下的嵌套目录层级**自由**，Kestra 只扫描所有子目录的 `flow.yaml`

**空目录处理**：
- Git 不跟踪空目录
- 新建文件夹时自动创建 `.gitkeep` 占位文件
- 每个文件夹下都放置 `.gitkeep`（统一逻辑，简化实现）

## 四、画布界面

### 4.1 布局（三栏）

| 区域 | 内容 | 默认宽度 |
|-----|------|---------|
| 左侧 | 项目树（本地仓库 `_flows/` 目录映射） | 240px，可拖拽调整 |
| 中间 | 画布（DAG 节点 + 连线） | 自适应 |
| 右侧 | 节点类型面板（SQL 节点、Shell 节点） | 默认展开 |

### 4.2 空画布状态

无 Flow 选中时，中间画布显示空状态 + 提示文字（"从左侧选择或新建 Flow"），不显示单独的欢迎页。

### 4.3 节点类型（MVP）

| 节点类型 | 说明 |
|---------|------|
| SQL 节点 | 双击弹出全屏编辑面板 → 选择数据源 → 写 SQL |
| Shell 节点 | 双击弹出全屏编辑面板 → 写 Shell 脚本 |

**数据源选择**：SQL 节点复用 wb-data 已配置的数据源（MySQL / PostgreSQL / Hive / StarRocks）。

### 4.4 连线规则

- **支持分支并行**：A → [B, C 并行] → D
- **不做条件分支**（SWITCH / if-else）
- **连线样式**：贝塞尔曲线（圆润平滑）
- **join 逻辑**：Kestra 的 `Parallel` 任务自动处理等待逻辑，wb-data 只需生成正确 YAML
- **禁止自连**：不能自己连自己（A → A）
- **禁止环**：不能形成环（A → B → C → A）

**环检测实现**：
- 使用 ReactFlow 的 `isValidConnection` 回调
- 用户连线时自动检测，阻止会形成环的连线
- 使用 `getOutgoers` 工具遍历图，通过递归检测是否存在环

**修改连线**：
- 支持拖拽线尾到目标节点，直接改变连接
- 也可以选中线按 `Delete` 删除后重新连接

### 4.5 DAG → Kestra YAML 编译

#### 示例场景

画布上有 4 个节点，连线关系如下：

```
     ┌── B ──┐
A →  │       │ → D
     └── C ──┘
```

| 节点 | 类型 | 作用 | 脚本路径 |
|-----|------|------|---------|
| **A** | Shell | 初始化数据 | `scripts/data_pipeline/etl/run.sh` |
| **B** | SQL | 查询数据（与 C 并行） | `scripts/data_pipeline/etl/query.sql` |
| **C** | Shell | 转换数据（与 B 并行） | `scripts/data_pipeline/daily_report/transform.sh` |
| **D** | Shell | 导出结果 | `scripts/data_pipeline/daily_report/export.sh` |

当检测到 D 有两条入边来自 B 和 C，后端自动将 B 和 C 包入 `Parallel` 块。

画布上的 DAG 编译为 Kestra YAML 示例：

```yaml
tasks:
  - id: A
    type: io.kestra.plugin.scripts.shell.Commands
    namespaceFiles:
      enabled: true
      include:
        - scripts/data_pipeline/etl/run.sh
    commands:
      - chmod +x scripts/data_pipeline/etl/run.sh
      - bash scripts/data_pipeline/etl/run.sh

  - id: parallel_tasks
    type: io.kestra.plugin.core.flow.Parallel
    tasks:
      - id: B
        type: io.kestra.plugin.scripts.shell.Commands
        namespaceFiles:
          enabled: true
          include:
            - scripts/data_pipeline/etl/query.sql
        commands:
          - beeline -u jdbc:hive2://hive-server:10000 -f scripts/data_pipeline/etl/query.sql
      - id: C
        type: io.kestra.plugin.scripts.shell.Commands
        namespaceFiles:
          enabled: true
          include:
            - scripts/data_pipeline/daily_report/transform.sh
        commands:
          - bash scripts/data_pipeline/daily_report/transform.sh

  - id: D
    type: io.kestra.plugin.scripts.shell.Commands
    namespaceFiles:
      enabled: true
      include:
        - scripts/data_pipeline/daily_report/export.sh
    commands:
      - bash scripts/data_pipeline/daily_report/export.sh
```

### 4.6 画布操作栏

固定在画布内容顶部，不随画布滚动。

按钮从左到右：`[全选 ☐] [提交] [调度] [执行] [执行结果]`

| 按钮 | 功能 |
|-----|------|
| 全选 ☐ | checkbox，选中画布上所有节点 |
| 提交 | git commit 到本地仓库 |
| 调度 | 打开调度配置弹窗 |
| 执行 | 有选中节点则执行选中；无选中则提示"请选择要执行的节点" |
| 执行结果 | 打开执行结果弹窗 |

**执行逻辑**：
- 有节点被勾选 → 执行选中的节点
- 无节点被勾选 → 提示"请选择要执行的节点"
- 节点执行中 → 该节点的 checkbox 禁用，不可重复选

**未提交红点**：
- `提交` 按钮右上角显示红色小圆点，表示有未提交的改动
- 点「提交」成功后红点消失
- 刷新页面后重新检测本地仓库是否有未提交的改动，有则重新出现

### 4.7 项目树顶部布局

```
[main]  [🔄 推送]
```

- 分支名称（MVP 阶段纯文本显示 `main`，无下拉框）
- 🔄 推送按钮（仅项目管理员可见）

普通用户只看到分支名称，没有推送按钮。

## 七、执行功能

### 7.1 执行逻辑

| 情况 | 行为 |
|-----|------|
| 有节点被勾选 | 执行选中的节点 |
| 无节点被勾选 | 提示"请选择要执行的节点" |
| 节点执行中 | 该节点的 checkbox 禁用，不可重复选 |

### 7.2 节点执行实现方式

通过 Kestra 的 `disabled` flag 实现选中节点执行：

1. 用户勾选节点 B 和 C，点「执行」
2. 后端临时给未勾选的节点（A、D）加 `disabled: true`
3. 执行整个 Flow，Kestra 跳过被禁用的节点，只执行 B 和 C
4. 不检查节点间依赖关系，用户自行负责（如跳过 A 导致 C 缺少数据由用户排查）

### 7.3 执行结果弹窗

**入口**：点击画布操作栏「执行结果」按钮

**关闭方式**：右上角 X 按钮

**内容**：当前 Flow 的执行记录，默认显示最近 3 天。

**列表字段**：

| 列 | 说明 |
|---|---|
| 停止按钮 | 仅执行中的任务显示，完成/失败隐藏 |
| 节点名称 | 如 `etl_node_01` |
| 节点类型 | SQL / Shell，图标区分 |
| 进度条 | 绿色静止=完成；绿色动画=进行中；红色=失败 |
| 开始时间 | `yyyy-MM-dd HH:mm:ss`，UTC 存储，前端显示浏览器本地时区 |
| 结束时间 | `yyyy-MM-dd HH:mm:ss`，UTC 存储，前端显示浏览器本地时区 |
| 耗时 | 单位：秒 |
| 📄 日志 | 查看执行日志，hover 提示"查看日志"；点击打开日志弹窗 |

**顶部功能**：
- 「停止所有」按钮：停止所有执行中的任务
- 时间范围筛选：最近 3 天 / 最近 7 天 / 自定义

**自定义时间范围**：
- 允许用户选择任意起止时间
- 限制最大范围（如不超过 30 天）

**执行结果弹窗打开时**，画布上节点双击编辑无效（编辑禁用）。

### 7.4 日志弹窗

**入口**：执行结果列表中点击 📄 日志按钮

**关闭方式**：右上角 X 按钮

**布局**：

```
┌──────────────────────────────────────────────────┐
│ 节点：transform_sql  [SQL]                  [✕] │
├──────────────────────────────────────────────────┤
│ [🔄 刷新]  [📥 下载]                              │
├──────────────────────────────────────────────────┤
│                                                  │
│ [2026-04-12 10:00:01] INFO  - 开始执行 SQL        │

│ [2026-04-12 10:00:03] ERROR - 写入失败           │
│                                                  │
├──────────────────────────────────────────────────┤
│ 状态：✅ 完成    耗时：4s                          │
└──────────────────────────────────────────────────┘
```

| 功能 | 说明 |
|-----|------|
| 标题栏 | 节点名称 + 类型标签 + 关闭按钮 |
| 🔄 刷新 | 执行中的任务需手动刷新查看最新日志 |

## 九、时间存储与显示

**Kestra 行为**（调研结论）：
- Kestra 内部统一用 **UTC** 存储执行时间（`startDate`、`endDate` 带 `Z` 后缀）
- Schedule Trigger 的 `timezone` 字段只影响 cron 解析，不影响存储格式
- Settings > Language/Region 设置只影响 Kestra UI 显示，不影响执行时间

**wb-data 处理**：
- 后端：不做时区转换，直接存储 Kestra 返回的 UTC 时间
- 前端：根据**浏览器本地时区**自动转换显示
- 调度配置的时区：只用于 cron 解析，不影响执行结果的存储/显示格式

## 十、项目树

**入口一：项目树顶部按钮**

**本地仓库**：`/data/repos/wb-data-{project_group_id}`

```
/data/repos/wb-data-001/
├── .git/
├── .gitkeep                         ← 根目录占位文件
├── _flows/
│     ├── data_pipeline/
│     │     ├── .gitkeep             ← 隐藏不显示
│     │     ├── etl/
│     │     │     ├── .gitkeep
│     │     │     └── flow.yaml       ← namespace: pg-001
│     │     └── daily_report/
│     │           ├── .gitkeep
│     │           └── flow.yaml       ← namespace: pg-001
│     └── archive/
│           └── .gitkeep
└── scripts/
      └── data_pipeline/
            └── etl/
                  └── query.sql       ← flow.yaml 通过 namespaceFiles 引用
```

**界面显示（项目树）**：

```
📁 policy_data
  📁 data_pipeline
    📁 etl
      📄 etl
    📁 daily_report
      📄 daily_report
  📁 archive
```

**映射关系说明**：

| 本地路径 | 界面显示 | 说明 |
|---------|---------|------|
| `/data/repos/wb-data-001/` | 📁 policy_data | 仓库根目录映射为项目组名称 |
| `_flows/` | （隐藏） | Kestra 约定目录，用户不可见 |
| `_flows/data_pipeline/` | 📁 data_pipeline | 直接显示 |
| `_flows/data_pipeline/etl/` | 📁 etl | 直接显示 |
| `_flows/data_pipeline/etl/flow.yaml` | 📄 etl | 显示文件名（无扩展名）|
| `.gitkeep` | （隐藏） | 占位文件，不显示 |
| `scripts/` | 📁 scripts | scripts 目录显示在项目树下 |

**关键规则**：
- `_flows/` 目录对用户不可见
- `.gitkeep` 占位文件在项目树中隐藏不显示
- `flow.yaml` 扩展名不显示，只显示文件名（如 `etl` 而非 `etl/flow.yaml`）
- namespace 定义在 `flow.yaml` 文件内容中，与目录结构无关

**显示规则**：
- `.gitkeep` 占位文件在项目树中隐藏不显示

## 十一、错误处理

MVP 阶段采用**分级处理**原则，不做复杂错误恢复流程，但必须有明确的错误反馈。

| 错误类型 | 处理方式 | 说明 |
|---------|---------|------|
| **本地仓库连不上** | 全屏错误页 + 联系管理员 | 服务级故障，用户无法操作 |
| **GitHub 推送失败** | 弹窗提示具体原因（网络超时 / 认证失败 / 冲突） | 用户需要知道为什么推不上去 |
| **Kestra 执行失败** | 执行结果面板显示红色进度条 + 📄 查看日志 | 开发过程中常见，用户需要日志排错 |
| **localStorage 满了** | 提示"浏览器存储空间不足，请清理缓存" | 罕见但会发生 |
| **其他未知错误** | 通用错误提示 + 错误码（方便反馈给开发） | MVP 够用就行 |

**核心原则**：不让用户面对一片空白或转圈卡死，至少告诉用户：出事了、大概什么事、我能做什么。

## 十四、冲突检测机制

### 14.1 背景

由于所有操作都在 wb-data 后端服务器进行，不允许去 GitHub Web 操作，冲突场景简化为：多人同时打开同一个 Flow 编辑。

### 14.2 打开 Flow 时检测

**流程**：
1. 后端读取本地文件，记录 `lastModified`
2. 检查 localStorage 是否有该 Flow 的草稿
3. 如果无草稿 → 直接加载文件内容，记录 `fileUpdatedAt`
4. 如果有草稿：
   - `savedAt <= fileUpdatedAt` → 冲突弹窗
   - `savedAt > fileUpdatedAt` → 用本地草稿恢复

**冲突弹窗**：
```
检测到文件已被修改。
你的本地草稿基于旧版本编写。

[保持本地草稿]  [加载最新]
```

### 14.3 点"提交"时检测

**流程**：
1. 读取本地文件当前 `lastModified`
2. 比较：`lastModified == fileUpdatedAt`（打开时记录的值）？
   - **相等**：文件没变，直接提交
   - **不等**：冲突弹窗

**冲突弹窗**：
```
检测到文件已被修改。
你的修改可能与其他用户的修改冲突。

[强制提交]  [加载最新]  [取消]
```

| 选项 | 行为 |
|-----|------|
| 强制提交 | 用我的内容覆盖文件，提交 |
| 加载最新 | 放弃我的修改，加载最新文件 |
| 取消 | 什么都不做，回到编辑 |

### 14.4 未覆盖的边界（MVP 不做）

- 节点级并发编辑：同一 Flow 内多个节点被不同用户同时编辑
- A 和 B 同时编辑，A 先提交，B 的草稿处理

## 十五、本地草稿机制

### 15.1 背景

用户编辑 Flow 内容后，可能刷新页面、关闭页面、切换页面，导致编辑内容丢失。需要草稿机制保护用户输入。

### 15.2 方案概述

草稿分两层存储：
- **localStorage**：快速临时存储，每隔几秒自动保存
- **本地文件**：持久化存储，用户点「保存」时写入

### 15.3 存储设计

**存储位置**：浏览器 **localStorage**

**key 格式**：`wb-data:draft:{flow_path}`，例如 `wb-data:draft:pg-123/my-flow`

**value 结构**：
```json
{
  "content": "...",
  "savedAt": 1712820000000,
  "fileUpdatedAt": 1712819000000,
  "originalContent": "..."
}
```

| 字段 | 说明 |
|-----|------|
| content | 草稿内容（编辑器内的当前内容） |
| savedAt | 草稿最后保存时间（时间戳） |
| fileUpdatedAt | 打开编辑器时本地文件的最后修改时间 |
| originalContent | 打开编辑器时加载的原始内容（用于比对是否有修改） |

### 15.4 保存策略

**自动保存（每隔几秒）**：
- 编辑时**每 5 秒**检测一次内容是否有变化
- 有变化则写入 localStorage

**手动保存（用户点「保存」）**：
- 写入 localStorage
- 同时写入本地文件（持久化）
- 按钮旁显示"最后保存: HH:mm:ss"

**关闭页面时**：
- 检测是否有未保存的本地修改（`content` != `originalContent`）
- 有修改则在关闭前将当前内容写入 localStorage
- 不弹出任何确认框，用户无感知

**点「提交」成功**：
- 不清除 localStorage（草稿保护机制，刷新后还能恢复）
- 只有用户主动覆盖/放弃草稿时才清除

### 15.5 恢复流程

**打开 Flow 时**：
1. 后端读取本地文件，记录 `fileUpdatedAt`
2. 检查 localStorage 是否有该 Flow 的草稿
3. 如果无草稿 → 直接加载文件内容
4. 如果有草稿：
   - 比较 `savedAt` 和 `fileUpdatedAt`
   - `savedAt <= fileUpdatedAt`：草稿已过期，按"冲突处理"流程
   - `savedAt > fileUpdatedAt`：本地草稿更新，直接用草稿恢复

**冲突处理**：
当检测到草稿基于旧版本时，弹窗提示：

```
检测到文件已被修改。
你的本地草稿基于旧版本编写。

[保持本地草稿]  [加载最新]
```

- **保持本地草稿**：保留 localStorage 草稿内容，加载到编辑器
- **加载最新**：丢弃本地草稿，加载本地文件最新内容

### 15.6 各场景处理

| 场景 | 处理方式 |
|-----|---------|
| 用户修改后刷新页面 | 自动用草稿恢复 |
| 用户修改后关闭页面再打开 | 自动用草稿恢复 |
| 用户点「保存」 | 写入 localStorage + 本地文件 |
| 用户点「提交」 | git commit，不清除草稿 |
| 用户选择"加载最新" | 清除草稿，加载本地文件最新 |
| 其他用户修改了同一文件 | 冲突弹窗，用户二选一 |

## 十六、项目组状态管理

### 16.1 背景

当前 wb-data 项目组采用物理删除，存在数据安全隐患。离线开发模块引入后，项目组与 GitHub 仓库、Flow 调度深度绑定，需要更精细的状态管理。

### 16.2 状态设计

```java
status: ACTIVE | INACTIVE | DELETED
deleted_at: LocalDateTime  // 只有 DELETED 时才有值
```

| 状态 | 含义 | 能否恢复 | GitHub |
|-----|------|---------|-------|
| ACTIVE | 正常使用 | — | 正常 |
| INACTIVE | 停用（如项目暂停） | ✅ 可恢复 | archive（只读） |
| DELETED | 软删除 | ✅ 可恢复（到 INACTIVE） | archive（只读） |

**恢复路径**：DELETED → INACTIVE → ACTIVE（不直接变 ACTIVE，需先确认）

### 16.3 INACTIVE 行为

当项目组从 ACTIVE 变为 INACTIVE 时：
- 该项目组下所有 Flow 的定时调度**立即停止**
- GitHub 仓库设置为 archive（只读）
- Kestra 侧同步暂停所有该 namespace 下的 Flow 执行

**停用确认流程**：
```
用户点击"停用项目组"
→ 弹窗警告："停用后该项目组下所有 Flow 的定时调度将停止，已在执行的任务会继续完成"
→ 用户确认
→ 后端：批量更新 Kestra Flow 状态为 PAUSE
→ wb-data status → INACTIVE
```

### 16.4 DELETED 行为

- GitHub 仓库 archive（只读）
- wb-data 项目组记录软删除（`status: DELETED`，`deleted_at` 记录时间）
- 不做物理删除，数据可恢复

### 16.5 未覆盖的边界（MVP 不做）

- 彻底删除（二次确认 + 物理删除）是否需要
- 定时调度补跑机制（Flow 停用期间漏跑的调度如何处理）

## 十八、讨论纪要

### 本地仓库架构讨论（2026-04-12）

- **新增 wb-data 后端服务器本地仓库**：所有用户操作在本地仓库进行，不直接操作 GitHub
- **新建项目组时**：同时创建 GitHub 仓库 + clone 到本地（此时仓库为空，很快）
- **推送到远程**：仅项目管理员可见「推送到远程」按钮，普通用户只有「提交」到本地
- **约束**：所有操作只能在 wb-data 界面进行，不允许去 GitHub Web 界面操作

### 空目录处理讨论（2026-04-12）

- **问题**：Git 不跟踪空目录，新建文件夹后不往里放 Flow，刷新后文件夹会消失
- **方案**：每个文件夹下放置 `.gitkeep` 占位文件
- **结论**：统一逻辑，每个文件夹下都放 `.gitkeep`，简化实现

### 本地草稿讨论（2026-04-12）

- **方案**：localStorage 存草稿 + 本地文件持久化
- **自动保存**：每隔几秒写入 localStorage
- **手动保存**：用户点「保存」→ 写入 localStorage + 本地文件
- **提交不清除草稿**：保持保护机制，刷新后还能恢复

### 目录结构讨论（2026-04-10）

- 用户曾提出将脚本和 `flow.yaml` 放在同一任务目录下（而不是 `_flows/` 和 `scripts/` 分开）
- 经讨论，**采用官方约定的 `_flows/` + `scripts/` 双目录结构**，更规范且便于未来扩展共享脚本
- GitHub 嵌套目录完全支持，Kestra 的 `includeChildNamespaces: true` 可递归扫描
- namespace 定义在 `flow.yaml` 内部，与目录路径无关

### GitHub 仓库策略讨论（2026-04-11）

- **采用一库一项目组方案**：每个项目组有独立 GitHub 仓库 `wb-data-{project_group_id}`
- 仓库归属系统级机器人账号（如 `wb-data-bot`），所有代码操作通过该账号执行
- 项目组停用/删除时，GitHub 仓库做 archive 处理，不做物理删除

### 冲突检测机制讨论（2026-04-12）

- **改为和本地仓库比**：由于操作都在本地进行，冲突检测改为本地草稿 vs 本地文件
- **冲突检测时机**：打开 Flow 时检测草稿是否过期 + 点提交时检测文件是否变化
- **冲突选项**：打开时（保持草稿/加载最新），提交时（强制提交/加载最新/取消）

### 定时调度讨论（2026-04-12）

- **调度配置入口**：画布顶部操作栏
- **调度粒度**：Flow 级别（每个 Flow 可设置自己的定时调度）
- **调度配置 UI**：预设（月/周/天/小时）→ Cron Tab 切换
- **预设默认值**：天周期
- **参数组**：MVP 暂不做，参数替换涉及执行层改动较大，可后续迭代

### 节点编辑界面讨论（2026-04-12）

- **视图切换方式**：双击节点后，画布内容切换为编辑表单（不是模态框，不是路由变化）
- **按钮布局**：取消/保存按钮放在顶部靠右，数据源选择框与按钮同行
- **按钮设计**：图标按钮 + hover 提示，取消在左，保存在右
- **编辑器**：复用现有查询页的 Monaco Editor + warm-parchment 主题

### UI/交互讨论（2026-04-12）

- **欢迎页取消**：用户进入离线开发时，无 Flow 选中则显示空画布 + 提示文字，不需要单独的欢迎页
- **节点添加方式**：拖拽从右侧面板到画布（不支持双击快速添加）
- **节点外观**：简洁圆角矩形，左侧图标类型 + 右侧名称 + 最左 checkbox
- **执行结果**：弹窗形式，当前 Flow 最近 3 天记录，每条显示节点名称、类型、进度条、开始/结束时间、耗时
- **停止按钮**：执行中显示「停止」；顶部显示「停止所有」；完成/失败隐藏
- **新建 Flow 入口**：项目树顶部按钮 + 文件夹右键菜单，弹窗内容根据入口不同
- **执行结果时间范围**：最近 3 天 / 最近 7 天 / 自定义
