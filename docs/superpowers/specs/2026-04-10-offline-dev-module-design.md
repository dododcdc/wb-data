## 一、核心定位

离线开发模块是 wb-data 的任务编排平台，用于可视化编排和执行 Kestra Flow。
用户通过拖拽式 DAG 画布构建任务流程，文件存储于 Gitea，执行引擎为 Kestra。

---

## 二、技术架构

| 组件 | 角色 |
|-----|------|
| Gitea | Git 仓库，**唯一真相源**，存储 Flow 定义和脚本文件 |
| Kestra | 执行引擎，从 Gitea SyncFlows 同步后执行任务 |
| wb-data | 用户界面 + 项目组管理，不维护 Flow 文件副本 |

### Gitea 仓库策略

**一库一项目组**：每个项目组有独立 Gitea 仓库，仓库命名 `wb-data-{project_group_id}`，UI 显示项目组名称。

| 层级 | 命名方式 |
|-----|---------|
| namespace | `pg-{project_group_id}` |
| Gitea 仓库 | `wb-data-{project_group_id}` |
| UI 显示 | 项目组名称（如"数据平台组"） |

**机器人账号**：所有代码操作通过系统级机器人账号（如 `wb-data-bot`）执行，不依赖个人账号。

## 三、目录结构

使用 Kestra 官方约定的目录结构：

```
📁 Gitea 仓库
│
├── 📁 _flows/
│     └── 📁 {namespace}/
│           ├── 📁 {任务A}/
│           │     └── flow.yaml      ← Kestra 读的 Flow 定义
│           └── 📁 {任务B}/
│                 └── flow.yaml
│
└── 📁 scripts/
      ├── 📁 {任务A}/
      │     ├── query.sql
      │     └── transform.sh
      └── 📁 {任务B}/
            └── query.sql
```

**说明**：
- `_flows/` 是 Kestra SyncFlows 的扫描根目录（约定，不可更改）
- `scripts/` 存放 Flow 引用的脚本文件，可与 `_flows/` 下的任务目录结构对应
- namespace 在 `flow.yaml` 内部定义（如 `namespace: pg-{project_group_id}`）
- `_flows/` 下的嵌套目录层级**自由**，Kestra 只扫描所有子目录的 `flow.yaml`
- wb-data 左侧项目树直接映射 Gitea 的 `_flows/` 目录结构

## 四、画布界面

### 4.1 布局（三栏）

| 区域 | 内容 |
|-----|------|
| 左侧 | 项目树（Gitea `_flows/` 目录映射） |
| 中间 | 画布（节点 + DAG 连线） |
| 右侧 | 节点类型面板 |

### 4.2 节点类型（MVP）

| 节点类型 | 说明 |
|---------|------|
| SQL 节点 | 双击弹出全屏编辑面板 → 选择数据源 → 写 SQL |
| Shell 节点 | 双击弹出全屏编辑面板 → 写 Shell 脚本 |

**数据源选择**：SQL 节点复用 wb-data 已配置的数据源（MySQL / PostgreSQL / Hive / StarRocks）。

### 4.3 节点编辑界面

**全屏切换视图**（双击节点后画布切换到编辑视图，点击取消/保存后返回画布）。

**布局**：
```
┌──────────────────────────────────────────────────────────┐
│ 数据源：[▼ 选择数据源      ]         [✕ 取消] [💾 保存] │
├──────────────────────────────────────────────────────────┤
│                                                          │
│ ┌──────────────────────────────────────────────────────┐ │
│ │ SELECT * FROM table_a;                               │ │
│ │                                                      │ │
│ └──────────────────────────────────────────────────────┘ │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**按钮设计**：图标按钮 + hover 提示，取消在左，保存在右

| 按钮 | 提示文字 | 说明 |
|-----|---------|------|
| ✕ 取消 | 取消 | 不保存，关闭编辑面板，返回画布 |
| 💾 保存 | 保存 | 保存节点配置，关闭编辑面板，返回画布 |

### 4.4 连线规则

- **支持分支并行**：A → [B, C 并行] → D
- **不做条件分支**（SWITCH / if-else）
- **连线样式**：折线（直角），自动避开节点
- **join 逻辑**：Kestra 的 `Parallel` 任务自动处理等待逻辑，wb-data 只需生成正确 YAML

### 4.5 DAG → Kestra YAML 编译

画布上的 DAG 编译为 Kestra YAML 示例：

```yaml
tasks:
  - id: A
    type: io.kestra.plugin.scripts.shell.Commands
    commands:
      - bash {{ namespaceFiles['scripts/taskA/run.sh'] }}

  - id: parallel_tasks
    type: io.kestra.plugin.core.flow.Parallel
    tasks:
      - id: B
        type: io.kestra.plugin.hive.queries.Query
        datasourceId: "{{ inputs.hive_ds }}"
        sql: "SELECT * FROM table_a;"
      - id: C
        type: io.kestra.plugin.scripts.shell.Commands
        commands:
          - bash {{ namespaceFiles['scripts/taskA/transform.sh'] }}

  - id: D
    type: io.kestra.plugin.scripts.shell.Commands
    commands:
      - bash {{ namespaceFiles['scripts/taskA/export.sh'] }}
```

当检测到 D 有两条入边来自 B 和 C，自动将 B 和 C 包入 `Parallel` 块。

### 4.6 画布操作栏

画布顶部新增一行操作栏，承载 Flow 级别的操作按钮：

```
┌─────────────────────────────────────────────────────┐
│  画布顶部操作栏                                       │
│  [↑ 上线]  [⏰ 设置调度]  [▶ 手动触发]               │
├─────────────────────────────────────────────────────┤
│                                                     │
│                   画布内容                           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**按钮设计**：图标按钮 + hover 提示文字

| 图标 | 提示文字 | 说明 |
|-----|---------|------|
| ↑ 上线 | 上线 | push 到 Gitea |
| ⏰ 设置调度 | 设置调度 | 打开调度配置弹窗（cron 表达式） |
| ▶ 手动触发 | 立即执行 | 不依赖定时，立即执行一次 Flow |

**调度配置弹窗**：
- cron 表达式输入框
- 时区选择
- 保存 / 取消

## 五、Git 操作

| 操作 | 触发时机 | 实现方式 |
|-----|---------|---------|
| 读取 | 打开 Flow | 后端调 Gitea API 拉取 `_flows/{任务名}/flow.yaml` |
| 上线 | 点"上线" | wb-data 调 Gitea API 执行 add + commit + push |
| SyncFlows | push 后自动触发 | Kestra 官方插件处理，非 wb-data 实现 |

**上线流程**：填写 commit message → 确认 → Gitea add + commit + push → push 成功触发 Kestra SyncFlows → Kestra 同步 Flow 到执行环境。

## 六、冲突检测机制

**日期**：2026-04-12
**状态**：已确认，待实现

### 6.1 背景

去除 Flow 级别锁，改为通过本地草稿 + 冲突检测处理多人协作问题。任何人都可以打开任何 Flow 编辑，每人维护自己的本地草稿。点上线时和打开时都会检测冲突。

### 6.2 打开 Flow 时检测

**流程**：
1. 后端请求 Gitea API 拉取最新内容 + `lastModified`
2. 检查 localStorage 是否有该 Flow 的草稿
3. 如果无草稿 → 直接加载远程最新内容，记录 `remoteUpdatedAt`
4. 如果有草稿：
   - `savedAt <= remoteUpdatedAt` → 冲突弹窗
   - `savedAt > remoteUpdatedAt` → 用本地草稿恢复

**冲突弹窗**：
```
检测到远程文件已更新。
你的本地草稿基于旧版本编写。

[保持本地草稿]  [加载远程最新]
```

### 6.3 点"上线"时检测

**流程**：
1. 请求 Gitea 获取远程最新 `lastModified`
2. 比较：`lastModified == remoteUpdatedAt`（打开时记录的值）？
   - **相等**：远程没变，直接 push
   - **不等**：冲突弹窗

**冲突弹窗**：
```
检测到远程文件已被更新。
你的修改可能与其他用户的修改冲突。

[强制上线]  [加载远程最新]  [取消]
```

| 选项 | 行为 |
|-----|------|
| 强制上线 | 用我的内容覆盖远程（`git push --force`） |
| 加载远程最新 | 放弃我的修改，加载远程最新 |
| 取消 | 什么都不做，回到编辑 |

### 6.4 未覆盖的边界（MVP 不做）

- 节点级并发编辑：同一 Flow 内多个节点被不同用户同时编辑
- A 和 B 同时编辑，A 先上线，B 的草稿处理

## 七、本地草稿机制

**日期**：2026-04-11
**状态**：已确认，待实现

### 7.1 背景

MVP 方案中，用户在编辑器内修改 Flow 内容后无"保存"动作，直接点"上线"才 push 到 Gitea。这导致两个问题：

1. 用户编辑后未点上线就关闭页面或刷新浏览器，内容丢失
2. 与 Gitea Web 编辑器的体验对齐（无草稿），但用户体验风险高

为解决上述问题，引入本地草稿机制。

### 7.2 方案概述

在浏览器端（localStorage）保存用户编辑过程中的草稿内容，草稿不 push 到 Gitea，仅作为本地备份。点"上线"成功后自动清除草稿。

### 7.3 存储设计

**存储位置**：浏览器 **localStorage**

**key 格式**：`wb-data:draft:{flow_path}`，例如 `wb-data:draft:pg-123/my-flow`

**value 结构**：
```json
{
  "content": "...",
  "savedAt": 1712820000000,
  "remoteUpdatedAt": 1712819000000,
  "originalContent": "..."
}
```

| 字段 | 说明 |
|-----|------|
| content | 草稿内容（编辑器内的当前内容） |
| savedAt | 草稿最后保存时间（时间戳） |
| remoteUpdatedAt | 打开编辑器时远程文件的最后修改时间 |
| originalContent | 打开编辑器时加载的原始内容（用于比对是否有修改） |

### 7.4 保存策略

**自动保存**：
- 编辑时**每 5 秒**检测一次内容是否有变化
- 有变化则写入 localStorage

**手动保存**：
- 编辑器提供"保存草稿"按钮（位于状态栏）
- 用户可随时主动触发保存
- 按钮旁显示"最后保存: HH:mm:ss"

**关闭页面时**：
- 检测是否有未上线的本地修改（`content` != `originalContent`）
- 有修改则在关闭前将当前内容写入 localStorage
- 不弹出任何确认框，用户无感知

**点上线成功**：
- 清除 localStorage 中该 Flow 的草稿

### 7.5 恢复流程

**打开 Flow 时**：
1. 后端请求 Gitea API 拉取最新文件，记录 `remoteUpdatedAt`
2. 检查 localStorage 是否有该 Flow 的草稿
3. 如果无草稿 → 直接加载远程最新内容
4. 如果有草稿：
   - 比较 `savedAt` 和 `remoteUpdatedAt`
   - `savedAt <= remoteUpdatedAt`：草稿已过期，按"冲突处理"流程
   - `savedAt > remoteUpdatedAt`：本地草稿更新，直接用草稿恢复

**冲突处理**：
当检测到草稿基于旧版本时，弹窗提示：

```
检测到远程文件已更新。
你的本地草稿基于旧版本编写。

[保持本地草稿]  [加载远程最新]
```

- **保持本地草稿**：保留 localStorage 草稿内容，加载到编辑器，覆盖远程版本（用户后续可继续编辑后上线）
- **加载远程最新**：丢弃本地草稿，加载远程最新内容到编辑器

### 7.6 各场景处理

| 场景 | 处理方式 |
|-----|---------|
| 用户修改后刷新页面 | 自动用草稿恢复 |
| 用户修改后关闭页面再打开 | 自动用草稿恢复 |
| 用户 A 修改期间，用户 B 上线了新版本，用户 A 刷新 | 冲突弹窗，用户二选一 |
| 用户点上线成功 | 清除本地草稿 |
| 用户关闭页面不点上线 | 草稿保留，下次打开可恢复 |
| 用户选择"加载远程最新" | 清除本地草稿，加载远程版本 |

### 7.7 UI 设计

**状态栏位置（编辑器底部）**：`[💾 保存草稿]  最后保存: 14:28:03`

**冲突弹窗**：
- 标题：**检测到冲突**
- 正文：检测到远程文件已更新。你的本地草稿基于旧版本编写。
- 按钮：**保持本地草稿**（左侧）  **加载远程最新**（右侧）

### 7.8 未覆盖的边界（MVP 不做）

1. **节点级并发编辑**：同一 Flow 内多个节点被不同用户同时编辑
2. **A 和 B 同时开草稿，A 先上线**：B 的草稿如何处理（需手动解决冲突）

### 7.9 与原设计的关系

- 不改变"无保存动作"的核心理念
- 不破坏 Gitea 作为"唯一真相源"的原则
- 草稿仅存在于用户浏览器，不影响其他用户
- 点"上线"才涉及 Gitea 操作

## 八、项目组状态管理

**日期**：2026-04-11
**状态**：已确认，待实现

### 8.1 背景

当前 wb-data 项目组采用物理删除，存在数据安全隐患。离线开发模块引入后，项目组与 Gitea 仓库、Flow 调度深度绑定，需要更精细的状态管理。

### 8.2 状态设计

```java
status: ACTIVE | INACTIVE | DELETED
deleted_at: LocalDateTime  // 只有 DELETED 时才有值
```

| 状态 | 含义 | 能否恢复 | Gitea |
|-----|------|---------|-------|
| ACTIVE | 正常使用 | — | 正常 |
| INACTIVE | 停用（如项目暂停） | ✅ 可恢复 | archive（只读） |
| DELETED | 软删除 | ✅ 可恢复（到 INACTIVE） | archive（只读） |

**恢复路径**：DELETED → INACTIVE → ACTIVE（不直接变 ACTIVE，需先确认）

### 8.3 INACTIVE 行为

当项目组从 ACTIVE 变为 INACTIVE 时：
- 该项目组下所有 Flow 的定时调度**立即停止**
- Gitea 仓库设置为 archive（只读）
- Kestra 侧同步暂停所有该 namespace 下的 Flow 执行

**停用确认流程**：
```
用户点击"停用项目组"
→ 弹窗警告："停用后该项目组下所有 Flow 的定时调度将停止，已在执行的任务会继续完成"
→ 用户确认
→ 后端：批量更新 Kestra Flow 状态为 PAUSE
→ wb-data status → INACTIVE
```

### 8.4 DELETED 行为

- Gitea 仓库 archive（只读）
- wb-data 项目组记录软删除（`status: DELETED`，`deleted_at` 记录时间）
- 不做物理删除，数据可恢复

### 8.5 未覆盖的边界（MVP 不做）

- 彻底删除（二次确认 + 物理删除）是否需要
- 定时调度补跑机制（Flow 停用期间漏跑的调度如何处理）

## 十、讨论纪要

### 目录结构讨论（2026-04-10）

- 用户曾提出将脚本和 `flow.yaml` 放在同一任务目录下（而不是 `_flows/` 和 `scripts/` 分开）
- 经讨论，**采用官方约定的 `_flows/` + `scripts/` 双目录结构**，更规范且便于未来扩展共享脚本
- Gitea 嵌套目录完全支持，Kestra 的 `includeChildNamespaces: true` 可递归扫描
- namespace 定义在 `flow.yaml` 内部，与目录路径无关

### Gitea 仓库策略讨论（2026-04-11）

- **采用一库一项目组方案**：每个项目组有独立 Gitea 仓库 `wb-data-{project_group_id}`
- 仓库归属系统级机器人账号（如 `wb-data-bot`），所有代码操作通过该账号执行
- 项目组停用/删除时，Gitea 仓库做 archive 处理，不做物理删除

### 冲突检测机制讨论（2026-04-12）

- **去除 Flow 级别锁**：原设计有锁机制，但用户反馈锁状态管理复杂，且多人协作场景分析后，锁并不能完全防止冲突
- **改为冲突检测方案**：任何人可编辑任何 Flow，通过本地草稿 + 冲突检测处理并发问题
- **冲突检测时机**：打开 Flow 时检测草稿是否过期 + 点上线时检测远程是否变化
- **冲突选项**：打开时（保持草稿/加载远程），上线时（强制上线/加载远程/取消）

### 定时调度讨论（2026-04-12）

- **调度配置入口**：画布顶部操作栏，放置三个图标按钮（上线、设置调度、手动触发）
- **调度粒度**：Flow 级别（每个 Flow 可设置自己的定时调度）
- **调度配置 UI**：点击"设置调度"按钮弹出配置弹窗，包含 cron 表达式输入框、时区选择
- **参数组**：MVP 暂不做，参数替换涉及执行层改动较大，可后续迭代
