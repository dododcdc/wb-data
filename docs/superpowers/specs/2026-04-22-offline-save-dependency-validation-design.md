# 离线开发保存期依赖关系校验设计

**日期**：2026-04-22  
**状态**：已确认，待写实施计划  
**范围**：`wb-data-frontend` / `wb-data-server` 离线开发 Flow 保存链路

---

## 1. 背景与问题

当前离线开发画布允许用户编辑 DAG，但保存时并不是把画布上的 `edges` 原样写回最终 YAML。

后端保存结构化 Flow 文档时，会按现有编译规则：

1. 先根据 `edges` 做拓扑排序
2. 再按 layer 分组
3. 同一 layer 的多个节点编译成 `Parallel`
4. 后续单节点会自动依赖前一个 layer 的全部出口

因此，某些画布上**未明确画出的依赖关系**，会在保存后被系统隐式补出。

典型例子：

- 画布显式关系：`a -> b`
- 新增 `c`，但不连任何线
- 保存后，系统会把 `a`、`c` 视作同一 layer，并把 `b` 视作下一 layer
- 重新解析后，结果会变成：`a -> b`、`c -> b`

这会导致两个问题：

1. 用户看到的画布依赖关系与保存后真实依赖关系不一致
2. 用户没有明确配置依赖线，却被系统自动补成新的执行前置关系

本次设计目标是：**只要保存会引入隐式依赖，就必须阻止保存并提醒用户先明确依赖关系。**

---

## 2. 设计目标

### 2.1 产品目标

1. 画布上的依赖线必须保持“所见即所得”
2. 用户未明确画出的依赖关系，不允许在保存时被系统静默补出
3. 一旦存在隐式依赖风险，必须阻止保存
4. 提示用户回到画布手动补全依赖线，但**无需列出具体是哪条线缺失**

### 2.2 工程目标

1. 前端在点击保存前即可发现问题，避免用户点保存后才知道失败
2. 后端继续保留权威兜底校验，避免绕过前端
3. 前后端校验语义保持一致，围绕“保存后是否会产生新的隐式边”展开
4. 改动聚焦保存校验，不顺带修改 DAG 编译模型本身

---

## 3. 备选方案

### 方案 A：只做后端保存期校验

保存请求提交到后端后，由后端判断是否会产生隐式依赖，若会则拒绝保存。

**优点**

- 后端最接近真实编译语义
- 实现集中，规则单一

**缺点**

- 用户点击保存后才收到阻断反馈
- 交互偏慢

### 方案 B：只做前端本地校验

前端保存前自行推导是否会产生隐式依赖，若有则直接阻止保存。

**优点**

- 交互最快
- 用户可立即得到反馈

**缺点**

- 后端缺少兜底
- 容易出现前后端规则漂移

### 方案 C（采用）：前端预检阻断 + 后端权威兜底

前端点击保存前先做一次本地预检；后端保存接口再做一次同语义校验。

**优点**

- 用户能即时收到反馈
- 后端可防止绕过前端
- 满足“前端后端都校验”的要求

**缺点**

- 需要在两端各落一份判断逻辑
- 需要明确并约束两端共享同一判定语义

---

## 4. 核心设计

### 4.1 校验定义

保存前需要回答一个问题：

> **当前画布的显式 `edges`，在经过现有 DAG 编译/回读语义后，是否会产生新的依赖边？**

如果答案是“会”，则认为当前 Flow 存在**依赖关系缺失**，禁止保存。

这里的“缺失”不是指图中完全没有路，而是：

- 用户没有在画布上明确画线
- 但按系统保存语义，这条依赖会在保存后出现

---

### 4.2 前端预检

前端在点击“保存 Flow”时，在现有节点内容、连线、布局校验之前或同一阶段内，增加一层依赖完整性预检。

判定规则：

1. 基于当前节点与显式 `edges` 构建 DAG
2. 复用后端当前编译语义的核心规则：
   - 拓扑排序
   - longest-path layer 分配
   - 相邻 layer 在回读时会形成“上一层全部出口 -> 下一层全部入口”的依赖集合
3. 计算“保存后将出现的依赖边集合”
4. 与用户当前显式 `edges` 做比较
5. 若保存后边集合严格多于当前显式边集合，则判定为依赖关系缺失

前端一旦发现缺失依赖：

- 直接阻止保存
- 不发起保存请求
- 保留当前草稿不变

---

### 4.3 后端兜底校验

后端在 `saveFlowDocument` 的 graph-save 路径中增加同语义校验。

判定方式与前端一致：

1. 根据请求中的 `nodes + edges` 推导保存后会形成的依赖边集合
2. 若存在保存后新增、但请求中未显式声明的依赖边，则拒绝保存

后端拒绝保存时：

- 返回明确的业务错误
- 不写入 `flow.yaml`
- 不写入脚本文件
- 不更新布局文件

后端仍然是最终权威，即使前端因 bug 或绕过未拦住，服务端也不能接受这种不完整依赖图。

---

### 4.4 提示策略

产品已确认：

- **只提醒存在依赖关系缺失**
- **不列出具体是哪条依赖线缺失**

推荐提示文案：

> 当前 Flow 存在未明确配置的依赖关系，请先在画布中补全依赖线后再保存。

提示要求：

1. 前端预检失败时立即显示该提示
2. 后端兜底失败时，前端展示同一类提示，不暴露隐式边明细
3. 提示语义聚焦“依赖线未明确配置”，而不是“系统自动补线”或“layer 编译”

---

## 5. 数据与接口影响

### 5.1 前端

新增一个纯函数级校验模块，用于：

1. 从当前 DAG 计算保存后推导边集合
2. 判断是否存在隐式新增边

该模块只服务于保存期校验，不修改已有 `draftSession`、节点编辑器、恢复稿逻辑。

### 5.2 后端

在现有 `saveFlowDocument` graph-save 路径增加校验步骤，不新增独立接口。

原因：

1. 保存本身已经具备完整上下文
2. 后端兜底天然属于保存事务的一部分
3. 本次目标是先补齐正确性，而不是扩展新的预校验 API

---

## 6. 测试策略

### 6.1 前端

增加纯函数测试，至少覆盖：

1. `a -> b`，`c` 独立时，判定为**存在依赖关系缺失**
2. `a -> b` 且 `c -> b` 已明确画出时，判定为**允许保存**
3. 无隐式新增边的普通线性 DAG，判定为**允许保存**

### 6.2 后端

增加服务层或工具层测试，至少覆盖：

1. 请求图为 `a -> b`，`c` 独立时，保存被拒绝
2. 请求图显式包含所有保存后依赖边时，保存通过
3. 拒绝保存时，不落任何文件副作用

---

## 7. 非目标

本次不处理以下问题：

1. 不改变当前 DAG -> YAML 的编译语义
2. 不引入“自动补全缺失依赖线”能力
3. 不在提示中高亮或枚举具体缺失依赖边
4. 不顺带重构整个 stage / layer 模型

---

## 8. 预期结果

改造完成后：

1. 用户若新增 `c` 节点但未明确画依赖线，且保存会让 `c` 影响其他节点，则保存会被阻止
2. 用户必须先在画布上明确补全依赖关系，再允许保存
3. 画布中看到的依赖关系与保存后真实语义保持一致
