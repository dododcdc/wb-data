# 前端认证与项目组接入开发步骤

更新时间：2026-04-01

本文档用于指导 `wb-data-frontend` 分阶段接入后端认证、项目组上下文和权限体系。目标是把当前“单用户 + mock 用户信息”的前端，平滑改造成“真实登录 + 项目组切换 + 权限驱动菜单”的版本。

本文档强调三件事：

- 每一步都应可单独开发和验证
- 每一步完成后都应人工验证再进入下一步
- 先把结构和交互跑通，再逐步补样式细节

## 一、当前前端现状

当前前端的关键现状如下：

- 路由只有首页、数据源管理、自助查询，见 [router/index.tsx](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/router/index.tsx#L26)
- 用户信息还是 mock 数据，见 [store/index.ts](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/store/index.ts#L12)
- 请求层还没有自动注入 `Authorization`，见 [request.ts](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/utils/request.ts#L8)
- 顶部布局目前只有 logo、导航和用户名，没有项目组切换器，见 [Layout.tsx](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/views/Layout.tsx#L83)

所以前端接入的本质工作是：

1. 让请求层支持登录态
2. 让应用启动时能拿到“当前用户 + 当前项目组 + 权限”
3. 让布局、路由和页面按权限工作
4. 让数据源和查询请求都带上认证信息和项目组上下文

## 二、后端接口基线

当前已经可用的认证和上下文接口：

- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`
- `GET /api/v1/auth/context`

当前已经接入认证的业务接口：

- `GET /api/v1/datasources`
- `GET /api/v1/datasources/{id}`
- `POST /api/v1/datasources`
- `PUT /api/v1/datasources/{id}`
- `DELETE /api/v1/datasources/{id}`
- `PATCH /api/v1/datasources/{id}/status`
- `POST /api/v1/datasources/test-connection`
- `POST /api/v1/datasources/{id}/test`
- `GET /api/v1/query/metadata/...`
- `POST /api/v1/query/execute/{dataSourceId}`
- `POST /api/v1/query/export/{dataSourceId}/tasks`
- `GET /api/v1/query/metadata/{dataSourceId}/dialect`

重要接口语义：

- 登录成功后返回 Bearer token
- `/api/v1/auth/context` 返回：
  - 当前用户
  - 是否系统管理员
  - 当前项目组
  - 可访问项目组列表
  - 当前项目组下的 permissions
- 数据源列表接口需要 `Authorization`，并且需要请求参数 `groupId`
- 新建数据源和新建测试连接需要 `groupId`

## 三、建议总步骤

建议拆成 6 步，每一步都可以单独开发、单独验证。

### 第 1 步：请求层支持认证

目标：

- 前端能统一存取 token
- 所有请求自动带 `Authorization: Bearer ...`
- 收到 401 时有统一处理

建议改动：

- [request.ts](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/utils/request.ts#L1)
- 新增一个 auth storage 工具，例如：
  - `src/utils/auth.ts`

建议实现：

- `localStorage` 保存 token
- 提供：
  - `getAccessToken()`
  - `setAccessToken()`
  - `clearAccessToken()`
- Axios request interceptor 自动加 header
- Axios response interceptor 识别 401：
  - 清 token
  - 清用户上下文 store
  - 跳转 `/login`

本步不改页面，只改基础设施。

验证点：

1. 手工写入 token 后，请求头能带上 `Authorization`
2. 不带 token 访问受保护接口时，后端返回 401
3. 401 发生后，本地 token 被清理

### 第 2 步：登录页

目标：

- 提供真实登录入口
- 登录成功后跳转到应用首页

建议新增：

- `src/views/Login.tsx`
- `src/views/Login.css`
- `src/api/auth.ts`

建议路由调整：

- 新增 `/login`
- 未登录访问业务页时跳登录页

页面结构建议：

- Logo / 产品名
- 标题：`登录 WB Data`
- 用户名输入框
- 密码输入框
- 登录按钮
- 错误提示区域
- 可选：开发环境提示文案，例如“开发环境默认账号见项目文档”

交互规则：

- 点击登录按钮后按钮进入 loading 态
- 登录成功：
  - 保存 token
  - 跳转 `/`
- 登录失败：
  - 表单区域内显示错误消息
  - 不清空用户名
  - 密码框可保留也可清空，建议清空

验证点：

1. `sys_admin / Dev123456!` 可登录
2. 错误用户名或密码显示错误
3. `user_disabled / Dev123456!` 登录被拒绝

### 第 3 步：应用启动时拉上下文

目标：

- 登录后应用知道“我是谁”
- 应用知道“我属于哪些项目组”
- 应用知道“当前项目组下有哪些权限”

建议新增：

- `src/types/auth.ts`
- `src/api/auth.ts` 里增加 `getAuthContext(groupId?)`
- 扩展 [store/index.ts](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/store/index.ts#L1)，不要再只存 `username + role`

建议 store 结构至少包含：

- `token`
- `user`
- `systemAdmin`
- `currentGroup`
- `accessibleGroups`
- `permissions`
- `isAuthenticated`

建议流程：

- 应用初始化时：
  - 如果没有 token，跳 `/login`
  - 如果有 token，调用 `/api/v1/auth/context`
- 拉取成功：
  - 写入 store
- 拉取失败：
  - 清 token
  - 跳 `/login`

验证点：

1. `sys_admin` 登录后能看到所有项目组
2. `ga_alpha` 登录后只看到 `alpha`
3. `multi_user` 登录后看到 `alpha` 和 `beta`
4. `user_no_group` 登录后 `currentGroup` 为空

### 第 4 步：项目组切换器

目标：

- 顶部导航支持切换当前项目组
- 切组后上下文和权限同步变化

建议修改：

- [Layout.tsx](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/views/Layout.tsx#L83)
- 可复用已有 `SimpleSelect` 或自定义一个项目组选择器组件

建议新增组件：

- `src/components/ProjectGroupSelect.tsx`

放置位置建议：

- 顶部导航右侧，用户头像左边

交互规则：

- 有多个可访问项目组时显示下拉选择器
- 只有一个项目组时显示只读标签也可以
- 无项目组时显示占位文本：`未加入项目组`
- 切换项目组时：
  - 顶部切换器进入 loading
  - 调 `/api/v1/auth/context?groupId=...`
  - 更新 store
  - 触发页面数据刷新

验证点：

1. `multi_user` 可以在 `alpha` / `beta` 间切换
2. 切换后 `permissions` 会变
3. 切换后数据源列表按新组刷新

### 第 5 步：菜单、页面守卫和空状态页

目标：

- 菜单不再固定写死
- 无权限页面不给进
- 无项目组用户看到空状态页

建议改动：

- [Layout.tsx](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/views/Layout.tsx#L94)
- [router/index.tsx](/Users/wenbin/Projects/wb-data/wb-data-frontend/src/router/index.tsx#L26)
- 新增：
  - `src/views/NoGroupState.tsx`
  - `src/views/Unauthorized.tsx`

菜单规则建议：

- 首页：始终可见
- 数据源管理：有 `datasource.read`
- 自助查询：有 `query.use`
- 后续成员管理：有 `member.read`
- 后续项目组设置：有 `group.settings`
- 系统管理：仅 `systemAdmin`

页面守卫规则：

- 未登录：跳登录页
- 无当前项目组但不是系统管理页：
  - 显示空状态页
- 有 token 但无权限：
  - 显示无权限页或重定向到首页

验证点：

1. `DEVELOPER` 看不到未来的成员管理/项目组设置入口
2. `GROUP_ADMIN` 可看到
3. `user_no_group` 登录后显示空状态页
4. `sys_admin` 即使没有成员关系，也不应该被空状态页卡死

### 第 6 步：业务页面接入

目标：

- 数据源页和查询页真正使用当前项目组和当前权限工作

建议改动：

- `src/api/datasource.ts`
- `src/api/query.ts`
- `src/views/DataSourceList.tsx`
- `src/views/DataSourceForm.tsx`
- `src/views/Query.tsx`

数据源页面接入规则：

- 列表请求带 `groupId`
- 创建数据源带 `groupId`
- 测试新建连接带 `groupId`
- 无 `datasource.write` 时：
  - 隐藏新建按钮
  - 隐藏编辑、删除、状态切换等操作

查询页面接入规则：

- 请求统一带 token
- 当前没有项目组时：
  - 不允许进入主查询工作区
- 当前无 `query.use`：
  - 不允许进入查询页
- 当前无 `query.export`：
  - 隐藏导出按钮

验证点：

1. 数据源列表只展示当前项目组数据
2. 切组后数据源列表会变
3. 查询页面在无权限时不可进入
4. 导出按钮只在有 `query.export` 时显示

## 四、详细交互规则

这一部分不讲代码，而讲用户行为和反馈。

### 1. 登录页交互

- 用户输入用户名和密码后点击登录
- 登录中：
  - 按钮禁用
  - 按钮文案改为 `登录中...`
- 登录失败：
  - 在表单下方显示红色错误文案
  - 保留用户名输入
- 登录成功：
  - 页面直接跳到首页，不额外弹 toast

### 2. 启动恢复登录态

- 如果本地有 token，应用启动时自动请求 `/api/v1/auth/context`
- 加载期间显示全屏或页面级 loading
- 恢复成功直接进入应用
- 恢复失败则跳登录页

### 3. token 失效

- 任意接口返回 401：
  - 清空 token
  - 清空用户上下文
  - 跳转登录页
  - 可选显示提示：`登录状态已失效，请重新登录`

### 4. 无项目组用户

- 登录成功后如果 `currentGroup = null` 且不是系统管理员：
  - 隐藏业务菜单
  - 显示空状态页
- 空状态页文案建议：
  - 标题：`你还未加入任何项目组`
  - 描述：`请联系系统管理员将你加入项目组后即可开始使用`

### 5. 系统管理员

- 系统管理员即使没有成员关系，也不应该进入“无项目组空状态”
- 系统管理员应能看到所有项目组
- 当前阶段如果还没做系统管理页：
  - 可先让系统管理员默认进入首页
  - 并通过项目组切换器进入具体组上下文

### 6. 项目组切换

- 切换时不需要整页白屏刷新
- 只刷新上下文和依赖数据
- 切换期间：
  - 选择器显示 loading
  - 页面骨架屏或轻量 loading 即可

### 7. 无权限

- 如果用户通过 URL 直接访问无权限页面：
  - 不要只靠菜单隐藏
  - 应显示明确状态

建议文案：

- 标题：`你没有访问此页面的权限`
- 描述：`如需访问，请联系项目组管理员调整权限`

## 五、页面结构与样式约定

这部分不做像素级视觉规范，只做足够开发的结构约定。

### 1. 登录页

建议结构：

- 居中卡片布局
- 左上或卡片顶部显示 `WB Data`
- 标题 + 副标题
- 两个输入框
- 一个主按钮
- 一个错误提示区域

样式方向建议：

- 简洁、稳重，不要过度装饰
- 优先复用现有主题变量和按钮样式
- 卡片宽度建议 380px 到 440px

### 2. 顶部导航

现有顶部导航保留，不做大改。

新增元素顺序建议：

- 左侧：logo + 主导航
- 右侧：项目组切换器 + 用户信息

项目组切换器样式建议：

- 宽度固定或最小宽度 180px
- 当前组名过长时截断
- 无项目组时显示灰色占位文本

### 3. 空状态页

建议结构：

- 居中图标
- 标题
- 描述
- 当前阶段不需要按钮

### 4. 无权限页

建议结构：

- 简单插图或图标
- 标题
- 说明文案
- 返回首页按钮

### 5. 页面级加载态

建议优先复用现有：

- `RouteLoadingPage`
- `TopProgressBar`
- 各业务页 skeleton

不要新增太多不同风格的 loading。

## 七、分步验证清单

### 步骤 1 验证

- token 能保存和读取
- Axios 自动带 `Authorization`
- 401 自动清理登录态

### 步骤 2 验证

- 正确账号可以登录
- 错误账号提示清晰
- 禁用账号无法登录

### 步骤 3 验证

- 页面启动时能恢复上下文
- `sys_admin` / `multi_user` / `user_no_group` 三类账号行为正确

### 步骤 4 验证

- 项目组切换器显示正确
- 切换后上下文更新正确

### 步骤 5 验证

- 菜单按权限显示
- 无项目组空状态正确
- 无权限访问提示正确

### 步骤 6 验证

- 数据源页面接入成功
- 查询页面接入成功
- 切组后数据跟着切

## 八、执行建议

建议采用下面节奏：

1. 先完成步骤 1
2. 你验证
3. 再做步骤 2
4. 你验证
5. 依次推进

不要一次把 6 步全做完再联调，否则问题会很难定位。

如果按这个节奏推进，明晚开始时最适合先做：

- **步骤 1：请求层支持认证**

这是所有后续工作的基础，而且改动范围最小，风险最低。
